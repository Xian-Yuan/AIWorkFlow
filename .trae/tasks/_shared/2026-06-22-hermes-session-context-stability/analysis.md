# Analysis: Hermes Session Context Stability

## Architecture Context

### System boundaries

- Durable profile source lives under `.trae/hermes/profiles/<profile>/config.overlay.yaml`.
- Runtime profile config lives under `.tools/hermes-worker/profiles/<profile>/config.yaml` and is generated by `.trae/scripts/sync-hermes-workflow.ps1 -Apply`.
- Session state lives in SQLite files under `.tools/hermes-worker/state.db` and `.tools/hermes-worker/profiles/<profile>/state.db`.
- Hermes agent compression logic lives under `.tools/hermes-worker/hermes-agent/agent/`.
- Hermes dashboard/session APIs live under `.tools/hermes-worker/hermes-agent/hermes_cli/` and `hermes_state.py`.

### Dependency map

- Profile overlays determine main model name, provider, context length, compression thresholds, and auxiliary compression provider.
- `sync-hermes-workflow.ps1` copies profile overlays into runtime profile config.
- Agent initialization reads runtime config and initializes context compression.
- Conversation loop and compressor decide whether to summarize, truncate, abort, or spawn a child session.
- Session DB records parent/child lineage, end reasons, message counts, and search/list behavior consumed by desktop UI.
- Tests under Hermes agent validate compressor continuity and web server session behavior.

### Data and state ownership

- Repository owns profile overlay files, sync scripts, tests, and diagnostic scripts.
- Hermes runtime owns generated profile config, logs, session DBs, and user `.env` files.
- This task may update generated runtime config through sync but must not mutate secrets or destructively cleanup real session content.
- Existing DB anomalies are evidence and can be diagnosed; deletion requires a separate explicit cleanup decision.

## Mature Solution Evidence

### Project-local evidence

- `skills/hermes-jinli-implementer/references/hermes-session-diagnosis.md` identifies three root causes: over-aggressive compression, session branching, and ghost sessions.
- Current global runtime config has `compression.threshold: 0.5`, `target_ratio: 0.2`, `protect_last_n: 20`, `abort_on_summary_failure: false`, and `auxiliary.compression.provider: auto`.
- Current profile source overlays still set `model.default: astron-code-latest`, while Ba Ba expects `glm-5.1`.
- Runtime implementer config already has safer compression than global config but still uses `abort_on_summary_failure: false`.
- Runtime planner config lacks explicit context/compression settings in the source overlay.
- Implementer profile DB contains 45 sessions, 5636 messages, 11 compression-ended sessions, multi-child compression parents, and true ghost sessions.
- Logs show context length detection failures for `astron-code-latest` and OpenRouter 401s for the global/default `z-ai/glm-5.1` path.
- Latest desktop logs show the main profile sends `glm-5.1` to the XF-Coding endpoint and receives `PathDomainError: Model Not Found`.
- The API server emits structured terminal failures as `run.failed`, while the desktop stream hook only handled `error`, allowing a partial streamed assistant bubble to be cleared without a persistent failure message.

### Official/framework evidence

- Hermes code already supports explicit `model.context_length` and tests assert explicit config context length wins over auto detection.
- Hermes web server already contains lineage-aware session search and latest-descendant endpoints, so the mature path is to extend existing lineage logic rather than inventing a parallel session model.
- Hermes SessionDB already provides empty-session count/delete APIs with safeguards for active/archived sessions.

### External mature references

- No external implementation is required for the first repair: the issue is local runtime integration and Hermes code behavior.
- If model/provider facts need external confirmation, use provider official docs only and record them in the verification report.

### Options compared

| Option | Source | Pros | Cons | Decision |
|---|---|---|---|---|
| Config-only repair | Existing sync/profile workflow | Low risk, fixes model/context drift and safer compression defaults | Does not fully solve duplicate/ghost session UI behavior | Partial, must be included |
| Code-only repair | Hermes session/compression internals | Can address ghost/dedupe behavior | Leaves profile drift and wrong context trigger timing | Rejected as incomplete |
| Layered repair | Local diagnosis + Hermes existing APIs | Addresses config, compression, session presentation, and diagnostics together | Requires more verification | Selected |
| Destructive DB cleanup | SQLite direct delete | Quickly removes current ghost rows | Risky if metadata is wrong; user did not authorize deletion | Rejected for this task |

### Rejected shortcuts

- Do not blame the Zhipu/free compression model alone.
- Do not disable compression entirely as the default mature fix.
- Do not set a fake huge context length to avoid compression.
- Do not delete sessions based only on `sessions.message_count=0`.
- Do not edit runtime config only and forget source overlays.
- Do not bypass tests because the bug appears in user-visible state.

### Selected mature path

Repair the durable Hermes configuration first, then add tested protections around session history presentation and compression failure behavior. The selected path keeps compression enabled but conservative, uses explicit context length to stabilize trigger timing, fails closed on summary failure, hides only true ghosts by default, dedupes compression lineages to the newest usable tip, and records diagnostics for existing DB anomalies.

For the recurrence where a reply flashes and disappears, repair both involved layers: pair the intended `z-ai/glm-5.1` model with the OpenRouter provider, and make the desktop stream hook treat terminal run/response failure events as turn-ending assistant errors.

## Acceptance Criteria

- AC01: `jinli-planner` and `jinli-implementer` profile overlays use the intended main model name, explicit context length, and consistent conservative compression defaults.
- AC02: Runtime profile configs match the repository overlays after sync.
- AC03: Compression is configured to preserve more recent context and fail closed when summary generation fails.
- AC04: Session listing/search behavior does not surface true ghost sessions by default and dedupes compression lineages to the latest usable tip.
- AC05: A non-destructive diagnostic command reports ghost sessions, multi-child compression lineages, and severe compression ratios without deleting real history.
- AC06: Focused Hermes tests for web session behavior and compressor continuity pass.
- AC07: Desktop stream handling persists terminal `run.failed` / `response.failed` events as assistant error messages, preserving any already-streamed assistant text.
- AC08: Task verification report maps evidence back to all acceptance criteria and records any remaining risk.
