# 虚幻多线程基础架构

## 概述

Unreal Engine 是一个多线程引擎，其架构包含多个专用线程和线程池系统。理解 UE 的线程架构是编写正确多线程代码的前置条件。

## UE 线程架构总览

UE 在启动时会创建以下主要线程：

| 线程 | 说明 |
|------|------|
| **GameThread（主线程）** | 执行游戏逻辑、蓝图、Actor Tick。所有 UObject 操作必须在主线程 |
| **RenderThread（渲染线程）** | 处理渲染命令提交。接收 GameThread 通过 ENQUEUE_RENDER_COMMAND 推送的数据 |
| **RHI Thread** | 处理底层图形 API（DX12/Vulkan/Metal）的提交 |
| **Audio Thread** | 音频处理 |
| **IO Threads** | 文件读取和资源加载 |
| **TaskGraph 线程池** | 通用任务调度系统，引擎内部和用户代码共享的线程池 |

### 渲染管线说明

GameThread 和 RenderThread 的工作存在 1-2 帧的延迟：

```
GameThread    → 准备渲染数据 → 准备数据 → ...
                   ↓ ENQUEUE_RENDER_COMMAND
RenderThread  →              → 执行渲染 → ...
```

这意味着 GameThread 修改的渲染数据不会立即生效，需要延迟 1-2 帧。例如：移动一个 StaticMeshComponent 的位置后，其实际渲染位置会在 1-2 帧后更新。

## 竞态条件（Race Condition）

当多个线程同时读写共享数据时，若至少有一个线程在执行写操作，且没有同步机制保护，就会产生竞态条件。竞态条件的表现是：
- 结果是**不确定的**——每次运行可能不同
- 难以调试和复现
- 可能导致崩溃、数据损坏或逻辑错误

解决方法：使用锁（FCriticalSection）、原子操作（FThreadSafeBool / std::atomic）等同步机制。

## ENQUEUE_RENDER_COMMAND 宏

用于从 GameThread 向 RenderThread 推送渲染命令：

```cpp
ENQUEUE_RENDER_COMMAND(MyRenderCommand)(
    [Parameter](FRHICommandList& RHICmdList)
    {
        // 在 RenderThread 执行
    });
```

注意：RenderThread 捕获的是数据的**副本**（通过值捕获），因为到 RenderThread 实际执行时，源数据可能已被 GameThread 修改。

## 数据安全模式

一种经典的多线程数据安全模式是：为每个线程维护一份数据的独立副本（镜像），在线程间同步时只传递变更的最小信息量。这种方式减少了锁的竞争，提高了并发程度。

## 进程 vs 线程

| | 进程 | 线程 |
|--|------|------|
| 地址空间 | 独立 | 共享（同一进程内） |
| 通信方式 | IPC（管道/套接字等） | 直接读写共享内存 |
| 资源开销 | 大（独立内存空间） | 小（共享堆） |
| 崩溃影响 | 不影响其他进程 | 影响整个进程 |

C++ 11 引入了 `std::thread` 作为跨平台线程标准库。

## 线程生命周期

线程的状态转换：`New → Runnable → Running → Blocked → Terminated`

- **New**：线程创建完成
- **Runnable**：已准备好可被调度
- **Running**：正在执行
- **Blocked**：等待资源（如锁、I/O）
- **Terminated**：执行完成

## 线程池（Thread Pool）概念

线程池的核心结构：

```
Pool → Queue（任务队列）→ Worker Threads（工作线程）
```

工作流程：
1. 任务被添加到 Pool 的任务队列
2. Worker Threads 从队列中取出任务执行
3. 执行完毕的 Worker 继续取下一个任务
4. 空闲线程被回收

**优先级队列**：线程池使用优先级队列，高优先级任务（VIP/SVIP）优先被取出执行。

### UE 的 TaskGraph 线程池

UE 的 TaskGraph 是一个全局线程池，被以下系统共享使用：
- 游戏逻辑的各种异步任务
- 物理引擎
- 音频系统
- 渲染相关计算
- 用户代码的 AsyncTask/ParallelFor

因为所有系统共享同一个线程池，任何一个系统长时间占用 TaskGraph 线程都会影响其他系统。这就是"阻塞主线程"类问题的根源。

## 并发（Concurrency）vs 并行（Parallelism）

| 概念 | 说明 |
|------|------|
| **并发（Concurrency）** | 多个任务在同一个 CPU 核上**交替执行**，看起来像是同时进行 |
| **并行（Parallelism）** | 多个任务在**多个 CPU 核**上同时执行 |

并发关注的是**任务结构**（如何分解），并行关注的是**执行效率**（如何加速）。并发的程序可以在单核上运行，并行必须依赖多核硬件。
