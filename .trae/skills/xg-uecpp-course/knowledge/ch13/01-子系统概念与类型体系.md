# 子系统概念与类型体系

## 什么是 Subsystem

Subsystem 是 Unreal Engine 内置的单例管理模式。一个 Subsystem 是一个具有**可控生命周期**的单例对象，由引擎自动创建和销毁，可在 C++ 和 Blueprint 中访问。

Subsystem 的设计目的：替代传统将大量逻辑塞入 GameInstance 的臃肿做法。通过将不同领域的逻辑拆分到独立的 Subsystem 中，代码结构更清晰，可维护性更高。

## Subsystem 的五种类型

| 类型 | 生命周期范围 | 获取方式 | 典型用途 |
|------|------------|---------|---------|
| **Engine Subsystem** | 引擎进程级别 | `GEngine->GetSubsystem<>()` | 全局初始化、资源管理器 |
| **Editor Subsystem** | 编辑器会话级别 | `GEditor->GetSubsystem<>()` | 编辑器工具（不可在游戏包中使用） |
| **GameInstance Subsystem** | GameInstance 生命周期（跨关卡） | `GetGameInstance()->GetSubsystem<>()` | **最常用**：游戏级逻辑、持久数据 |
| **World Subsystem** | 单个 World 级别 | `GetWorld()->GetSubsystem<>()` | 关卡级逻辑、世界状态管理 |
| **LocalPlayer Subsystem** | 单个 LocalPlayer 级别 | `GetLocalPlayer()->GetSubsystem<>()` | 玩家数据、输入设置 |

### GameInstance Subsystem（最常用）

这是实践中使用频率最高的 Subsystem 类型。GameInstance 本身是跨关卡持久存在的对象，其 Subsystem 同样具有跨关卡持久性。

**核心特性**：

- **跨关卡持久** — 切换关卡时，Subsystem 的数据不会丢失（区别于 GameMode/GameState/PlayerController/PlayerState，这些对象在关卡转换时会被销毁重建）
- **引擎自动管理生命周期** — 无需手动 NewObject()，引擎根据 Subsystem 类型自动创建和销毁
- **蓝图中可直接访问** — 在蓝图中输入 "Subsystem" 即可看到已注册的 Subsystem 节点

### Engine Subsystem

引擎进程级别的单例。在整个引擎运行期间存在，适合管理全局资源。

### Editor Subsystem

仅在编辑器会话中生效的 Subsystem。**注意**：`UnrealEd` 模块在游戏打包时不可用，因此 Editor Subsystem 的实现需要放在编辑器模块中，或者在游戏代码中使用 `WITH_EDITOR` 预处理器宏进行条件编译。

### World Subsystem

按 World 生命周期管理的 Subsystem。一个关卡对应一个 World，切换关卡时旧的 World Subsystem 被销毁，新的被创建。

### LocalPlayer Subsystem

按本地玩家管理的 Subsystem。适合存储每个玩家的独立配置（如按键设置、画面设置）。

## 生命周期

1. **自动创建** — 引擎根据 UCLASS 声明自动实例化
2. **ShouldCreateSubsystem** — 可条件控制是否创建（如只在服务端创建）
3. **Initialize** — 初始化回调
4. **运行时** — 正常使用
5. **Deinitialize** — 清理回调
6. **自动销毁** — 引擎自动回收
