# 横向专题：网络通信演进

## 概述

从第 25 章到第 32 章，课程沿着一条清晰的通信链路逐层深入：**HTTP 基础 → HTTP 高级 → WebSocket 客户端 → WebSocket 服务端 → TCP**。伴随协议升级，异步架构、认证方式、工程组织也在同步演进。

| 章节 | 案例 | 传输协议 | 异步模式 | 认证方式 | 代码组织 |
|------|------|---------|---------|---------|---------|
| 第 25 章 | HTTP 获取公网时间 | HTTP GET | UBlueprintAsyncActionBase | 无 | 单个 C++ 文件 |
| 第 26 章 | HTTP 登录注册服务器 | HTTP POST | 独立程序主循环 | MD5 令牌 | 独立程序 + 插件 |
| 第 27 章 | 摄像头活体检测 | HTTP POST Upload | UBlueprintAsyncActionBase | HMAC-SHA256 | 双插件分离 |
| 第 28 章 | 百度文心一言 | HTTP Stream | UBlueprintAsyncActionBase + 流式回调 | HMAC-SHA256 (BCE v1) | 独立 Link 插件 |
| 第 29 章 | 讯飞语音识别 | WebSocket (客户端) | UGameInstanceSubsystem + FRunnable | HMAC-SHA1 | 插件内多子系统 |
| 第 30 章 | 讯飞语音合成 | WebSocket (客户端) | UBlueprintAsyncActionBase | HMAC-SHA256 | 同插件 TTS 模块 |
| 第 31 章 | MMO 分布式服务器通信 | WebSocket (客户端+服务端) | Subsystem + Tick | 双 GUID | 自建 WSM 插件 |

## 一、传输协议演进

### 1.1 HTTP 阶段（第 25~28 章）

**请求-响应模型**：客户端发起请求，服务端返回响应，一次一问一答。

```cpp
// 第 25 章—最简洁的 HTTP GET
// 参考：XGSampleHttpTime.cpp
FHttpRequestRef Request = FHttpModule::Get().CreateRequest();
Request->SetURL(TEXT("https://www.baidu.com"));
Request->SetVerb(TEXT("GET"));
Request->OnProcessRequestComplete().BindUObject(this, &UXGSampleHttpTime::OnHttpTimeResponse);
Request->ProcessRequest();
```

**演进过程**：

| 版本 | 新增复杂度 | 关键类 |
|------|-----------|--------|
| 第 25 章（获取公网时间） | 基础 GET + JSON 响应解析 | FHttpModule, FHttpRequestRef, FHttpResponsePtr, FJsonObjectConverter |
| 第 26 章（登录注册服务器） | HTTP Server（独立程序）+ POST + MD5 令牌 + JSON 序列化 | FHttpServerModule, FHttpServerResponse, MD5 链式哈希 |
| 第 27 章（摄像头活体检测） | 文件上传（multipart/form-data）+ 媒体捕获 + Base64 | Media Framework, FImageUtils, HMAC-SHA256 签名 |
| 第 28 章（百度文心一言） | 流式传输（Server-Sent Events）+ 实时解析 | FHttpRequest::OnStreamReady, SSE 分隔符解析 |

### 1.2 WebSocket 阶段（第 29~31 章）

**全双工模型**：一次握手建立持久连接，双方随时发送消息。

```cpp
// 第 29 章—WebSocket 客户端连接
// 参考：XGSampleSTTSubsystem
FString WebSocketUrl = AssembleAuthUrl();  // HMAC-SHA1 签名 URL
WebSocket = FWebSocketsModule::Get().CreateWebSocket(WebSocketUrl);
WebSocket->OnConnected().AddUObject(this, &UXGSampleSTTSubsystem::OnSTTWSConnected);
WebSocket->OnMessage().AddUObject(this, &UXGSampleSTTSubsystem::OnSTTWSMessage);
WebSocket->Connect();
```

**演进过程**：

| 版本 | 新增复杂度 | 关键特性 |
|------|-----------|---------|
| 第 29 章（语音识别） | 客户端 + 状态机 + 音频采集 + FRunnable 消费线程 | 状态机（Ready/Init/WaitToServerClose），3s 音频帧 |
| 第 30 章（语音合成） | 客户端 + USoundWave 创建 + WAV 文件存储 | Base64 音频解码，三种 USoundWave 创建方式 |
| 第 31 章（MMO 分布式服务器） | **服务端 + 客户端双角色** + 多连接管理 + 心跳 | IWebSocket + Tick 检测，TMap 管理连接，双 GUID 消息路由 |

### 1.3 TCP 阶段（第 32 章）

**面向流模型**：基于 FSocket 的原始字节流通信，无 HTTP 或 WebSocket 的协议层封装。

```cpp
// 第 32 章—TCP 连接
// 参考：XGSampleEMailRunnable.cpp
FSocket* Socket = FTcpSocketBuilder(TEXT("XGEMailSocket"))
    .AsBlocking()
    .Build();

TSharedRef<FInternetAddr> Address = ISocketSubsystem::Get(PLATFORM_SOCKETSUBSYSTEM)->CreateInternetAddr();
ISocketSubsystem::Get(PLATFORM_SOCKETSUBSYSTEM)->GetAddressFromString(ServerIP, *Address);
Address->SetPort(ServerPort);

bool bConnected = Socket->Connect(*Address);
```

**SMTP 协议状态机**：TCP 之上实现应用层协议（纯字节流，无框架辅助）

```
220 → EHLO → 250 → AUTH LOGIN → 334 → 
Base64(Username) → 334 → Base64(Password) → 235 → 
MAIL FROM → 250 → RCPT TO → 250 → 
DATA → 354 → 邮件内容 + "\r\n.\r\n" → 250 → 
QUIT → 221
```

## 二、异步架构演进

从第 25 章到第 32 章，异步模式经历了 5 个阶段，每个阶段解决前一个阶段的局限性：

### 2.1 UBlueprintAsyncActionBase（第 25/27/28/30 章）

最常见的异步蓝图节点模式，核心接口：

```cpp
// 标准模板（参考：XGSampleHttpTime.h/第 25 章）
UCLASS()
class UXGSampleHttpTime : public UBlueprintAsyncActionBase
{
    GENERATED_BODY()

    UPROPERTY(BlueprintAssignable)
    FOnCompleted OnCompleted;

    virtual void Activate() override;

    UFUNCTION(BlueprintCallable, meta = (BlueprintInternalUseOnly = "true", HideThen = "true"))
    static UXGSampleHttpTime* GetHttpTime(const UObject* WorldContextObject);
};
```

**局限**：依赖 UObject 生命周期，不能脱离 GameInstance 单独运行。

### 2.2 独立程序主循环（第 26 章）

摆脱 GameThread Tick 束缚，使用独立程序模式：

```cpp
// 参考：XGSampleServer.cpp（第 26 章）
INT32_MAIN_INT32_ARGC_TCHAR_ARGV()
{
    // 1. 启动引擎
    GEngineLoop.PreInit(...);
    // 2. 初始化自定义配置
    Config = LoadConfig();
    // 3. 绑定 HTTP 端口
    HttpRouter->Bind(Config->GetPort());
    // 4. 主循环
    while (!bIsExit)
    {
        FPlatformProcess::Sleep(0.05f);
        // 处理请求回调
    }
    // 5. 关闭序列
    GEngineLoop.Exit();
}
```

**优势**：无 GC、无 GameThread 约束，适合纯服务端。
**局限**：不能直接复用 UBlueprintAsyncActionBase 模式。

### 2.3 Subsystem（第 29/31 章）

```cpp
// 第 29 章—UGameInstanceSubsystem 管理 WebSocket 生命周期
UCLASS()
class UXGSampleSTTSubsystem : public UGameInstanceSubsystem
{
    // 状态机 + WebSocket 管理 + 音频采集
};

// 第 31 章—双 Subsystem 管理客户端和服务端
UCLASS()
class UXGWSMClientSubsystem : public UGameInstanceSubsystem;
UCLASS()
class UXGWSMServerSubsystem : public UGameInstanceSubsystem;
```

### 2.4 FRunnable（第 29/32 章）

依赖独立线程的场景使用 FRunnable：

```cpp
// 第 29 章—音频消费线程
class FXGSampleConsumeVoiceRunnable : public FRunnable;
// 第 32 章—TCP 通信线程
class FXGSampleEMailRunnable : public FRunnable;
```

### 2.5 TSharedFromThis AsyncAction（第 32 章）

脱离 UObject 体系的异步行动模式：

```cpp
// 第 32 章—TSharedFromThis 模式
class FXGSampleEMailAsyncAction : public TSharedFromThis<FXGSampleEMailAsyncAction>
{
    TSharedPtr<FXGSampleEMailRunnable> RunnablePtr;

    void Start();
    void Cancel();
    void OnCompleted(bool bSuccess, const FString& Message);
};
```

**与 UBlueprintAsyncActionBase 的对比**：

| 维度 | UBlueprintAsyncActionBase | TSharedFromThis AsyncAction |
|------|--------------------------|---------------------------|
| 继承体系 | UObject | TSharedFromThis |
| 生命周期 | 受 GC 管理 | 智能指针管理 |
| 蓝图支持 | 原生支持 | 通过 BPLibrary + Subsystem 间接暴露 |
| 独立程序兼容 | 不兼容 | 兼容 |
| 资源释放 | RegisterWithGameInstance + ReleaseResources | 手动管理，先 Stop 线程再释放 Runnable |

## 三、认证方式演进

| 章节 | 认证方式 | 实现机制 | 复杂度 |
|------|---------|---------|--------|
| 第 25 章 | 无认证 | 公网接口，直接请求 | ★ |
| 第 26 章 | MD5 链式哈希 | 自定义 Token 生成 + Header 传递 + 密码哈希存储 | ★★ |
| 第 27 章 | HMAC-SHA256 | OpenSSL HMAC + Base64 + URL 签名 | ★★★ |
| 第 28 章 | HMAC-SHA256 (BCE v1) | HttpUtil::AppendHeader 自动注入 Header | ★★★ |
| 第 29 章 | HMAC-SHA1 | GenerateRequireParams URL 参数签名 | ★★ |
| 第 30 章 | HMAC-SHA256 | AssembleAuthUrl URL 参数签名 | ★★ |
| 第 31 章 | 双 GUID 消息路由 | 消息体携带 GUID，无外部认证 | ★ |
| 第 32 章 | AUTH LOGIN (Base64) | SMTP 协议内置，Base64 编码用户名/密码 | ★★ |

**规律**：对外部 API 的调用倾向于 HMAC 系列签名（SHA256/SHA1），自定义服务端倾向于自定义 Token（MD5 + GUID）。

## 四、代码组织演进

| 阶段 | 组织方式 | 示例 | 何时使用 |
|------|---------|------|---------|
| 单文件 | 单个 .h/.cpp 对 | 第 25 章：XGSampleHttpTime | 功能单一，< 100 行 |
| 工程分离 | 独立程序 + 游戏插件 | 第 26 章：XGSampleServer + XGSampleClient | 需要服务端独立运行 |
| 功能插件化 | 按功能拆分为独立插件 | 第 27 章：XGSamplePicture + XGSampleLink | 多个项目复用同一功能 |
| 插件内模块化 | 插件内拆分子系统/线程 | 第 29/31 章：XGSampleXFLink/XGSampleWSM | 单个功能包含多个子系统 |
| 全静态封装 | 脱离 UObject，纯 C++ 封装 | 第 32 章：XGSampleEMail | 需要脱离引擎运行 |

**插件创建的规律**：

- 新插件使用 `Blank` 模板，从零构建
- 插件命名遵循 `XGSample{功能名}` 约定
- 插件内按职责拆分子目录：AsyncAction/、Subsystem/、Thread/、Type/、Log/、Util/

## 五、选型对比

### 5.1 传输协议选型

| 场景 | 推荐协议 | 原因 |
|------|---------|------|
| 获取/提交数据 | HTTP | 简单直接，通用性强，防火墙友好 |
| 实时双向通信 | WebSocket | 低延迟，全双工，适合语音/聊天/推送 |
| 需要服务端主动推送 | WebSocket | HTTP 需要轮询或 Server-Sent Events |
| 自定义应用层协议 | TCP | 完全控制协议格式，最高性能 |
| 标准邮件/文件传输 | TCP + 应用层协议 | SMTP/FTP 等标准协议基于 TCP |
| 大模型 AI 对话 | HTTP 流式传输 | SSE 协议成熟，兼容性好 |

### 5.2 异步模式选型

| 场景 | 推荐模式 | 原因 |
|------|---------|------|
| 蓝图可用的一次性 HTTP 请求 | UBlueprintAsyncActionBase | 蓝图原生支持，使用简单 |
| 需要独立服务端 | 独立程序主循环 | 无 GC 开销，可脱离 Editor |
| 长连接管理（WebSocket） | UGameInstanceSubsystem | 与 GameInstance 生命周期绑定 |
| 耗时音频/数据消费 | FRunnable | 独立线程，不阻塞 TaskGraph |
| 可脱离引擎运行的异步任务 | TSharedFromThis AsyncAction | 无 UObject 依赖 |
| 多步骤异步编排 | ControlFlows | 声明式流程控制 |
