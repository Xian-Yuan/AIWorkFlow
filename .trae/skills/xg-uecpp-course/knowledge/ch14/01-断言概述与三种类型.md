# 断言概述与三种类型

## 什么是断言

断言（Assertion）是开发阶段用于检查"不应该发生的情况"的机制。当断言条件失败时，说明代码执行到了预期之外的状态，需要开发者介入排查。

断言**只在 Debug/Development 构建中生效**，Shipping 构建默认移除大部分断言代码。因此断言不能用于运行时错误处理（如玩家输入校验），只能用于开发阶段的不变量检查。

## 断言的核心头文件

```cpp
Engine/Source/Runtime/Core/Public/Misc/AssertionMacros.h
```

该文件定义了 UE 中所有断言宏。

## 三大断言类别

UE 提供了三个断言类别，各自的行为差异是关键：

| 类别 | 宏 | 失败时行为 | 可用构建 |
|------|-----|-----------|---------|
| **check** | check, checkf, checkNoEntry, checkCode, checkSlow | **停止执行**（触发断点/Crash） | Debug + Development（checkSlow 仅 Debug） |
| **verify** | verify | **停止执行**，但表达式在 Shipping 中仍执行 | 全部构建 |
| **ensure** | ensure, ensureMsgf | **记录第一次失败后继续执行** | Debug + Development + Shipping（行为可选） |

### check —— "最强力，最暴力"

- 条件失败时直接停止执行
- 在 Shipping 构建中完全移除（不编译）
- 适用场景：检查空指针、类型转换、数组越界等绝对不可能发生的情况

### verify —— "与 check 类似，但表达式不消失"

- 行为和 check 几乎一致（失败时停止执行）
- **关键差异**：verify 的参数表达式在 Shipping 构建中**仍然会被执行**，只是不检查结果
- 适用场景：条件需要产生副作用，且 Shipping 中这个副作用必须保留

### ensure —— "失败也能继续运行"

- 条件失败时记录日志，但允许程序继续执行
- **只记录第一次失败**，后续相同条件的 ensure 不再触发（避免日志刷屏）
- 返回布尔值，可用于流程控制
- 适用场景：可有可无的检查，失败不应导致崩溃（如 UI 更新、非关键资源加载）

## 构建配置对断言的影响

| 构建配置 | check 系列 | verify | ensure | checkSlow |
|---------|-----------|--------|--------|-----------|
| DebugGame | ✓ | ✓ | ✓ | ✓ |
| DebugEditor | ✓ | ✓ | ✓ | ✓ |
| Development | ✓ | ✓ | ✓ | ✗ |
| Shipping | ✗ | 表达式执行 | 可配置 | ✗ |

- **Shipping 无断言**：所有 check 系列代码在 Shipping 中完全不编译，这不是 Bug 而是设计
- **checkSlow 仅 Debug**：checkSlow 只在 DebugGame/DebugEditor 中运行，Development 也不执行

## 核心原则：不要把业务逻辑写到断言参数里

```cpp
// 错误！ModifyHealth() 在 Shipping 中不会执行
ensure(ModifyHealth());

// 正确：单独调用 ModifyHealth()，再断言结果
ModifyHealth();
check(Health > 0);
```

如果将带有副作用的函数调用放到断言参数中，Shipping 构建的行为会和 Debug 不一致，导致难以追溯的 Bug。
