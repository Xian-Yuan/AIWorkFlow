# 蓝图项目插件模块丢失修复

## 概述

纯蓝图项目（BlueprintOnly 项目）在引用带 C++ 模块的插件时，打包运行后会出现模块丢失的问题。这是因为 BlueprintOnly 项目在打包流程中不会编译 C++ 模块，导致运行时无法加载插件功能。

---

## 问题现象

- 插件在编辑器中正常工作
- 使用 **Package Project** 打包项目
- 运行打包后的可执行文件
- 插件功能失效，表现为模块未加载或功能丢失

---

## 原因分析

| 因素 | 说明 |
|------|------|
| **BlueprintOnly 项目** | 项目创建时未包含 C++ 源码，打包时不触发 C++ 编译流程 |
| **插件 C++ 模块** | 插件的 C++ 代码需要在打包时被编译和链接 |
| **运行时加载** | 打包后的可执行文件找不到插件模块的二进制文件 |

---

## 修复方案

### 步骤 1：修改 `.uplugin` 文件

在插件描述文件（`.uplugin`）中调整模块配置：

```json
{
    "FileVersion": 3,
    "FriendlyName": "插件名称",
    "Version": 1,
    "VersionName": "1.0",
    "Modules": [
        {
            "Name": "模块名",
            "Type": "Runtime",
            "LoadingPhase": "Default",
            "EnabledByDefault": true,
            "Installed": true
        }
    ]
}
```

### 关键字段说明

| 字段 | 值 | 作用 |
|------|----|------|
| `EnabledByDefault` | `true` | 插件默认启用，无需手动勾选 |
| `Installed` | `true` | 标记为已安装插件，打包时会被包含 |
| `LoadingPhase` | `Default` | 模块加载阶段，确保在正确时机加载 |

### 步骤 2：调整模块加载顺序

如果插件包含多个模块，需要调整加载顺序确保依赖模块先加载：

- 主功能模块设为 `Default` 或 `PreDefault`
- 依赖的基础模块设为 `EarliestPossible` 或 `PreDefault`
- 确保所有模块的 `EnabledByDefault` 和 `Installed` 均为 `true`

---

## 验证流程

### 完整验证步骤

1. **Package Plugin**：先打包插件本身，确认无编译错误
2. **Package Project**：将项目打包，观察是否有插件相关的警告或错误
3. **Run Packaged Build**：运行打包后的可执行文件，验证插件功能正常工作

### 验证清单

| 检查项 | 说明 |
|--------|------|
| 插件在编辑器中加载正常 | 打开编辑器确认插件列表显示已启用 |
| 插件功能在编辑器中可用 | 使用插件提供的功能，确认无报错 |
| Package Plugin 无错误 | 单独打包插件成功 |
| Package Project 无警告 | 项目打包成功，无模块丢失警告 |
| 打包后运行正常 | 运行打包构建，插件功能正常 |

---

## 注意事项

- 该修复方案适用于**需要给蓝图项目使用的插件**
- 如果插件只给 C++ 项目使用，不需要设置 `EnabledByDefault` 和 `Installed`
- 修改 `.uplugin` 后建议清除中间文件（`Intermediate/`、`Binaries/`）再重新打包
- `.uplugin` 文件中不能包含注释，修改后需检查 JSON 格式的合法性
