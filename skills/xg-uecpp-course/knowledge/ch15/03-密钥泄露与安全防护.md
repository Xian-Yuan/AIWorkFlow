# 密钥泄露与安全防护

## 问题的根源

当使用 DeveloperSettings 存储敏感信息（如 API Key、AppSecret）时，这些值以**明文**形式存储在 INI 文件中。打包后，INI 文件被嵌入 `.pak` 包中，用户可以直接从 pak 中提取 INI 文件读取其中的密钥。

这意味着：**任何存储在 Config 属性中的字符串，在发布后的产品中都是明文可见的**。

## 泄露风险演示

```cpp
UPROPERTY(Config, EditAnywhere, BlueprintReadWrite, Category = "XG")
FString XGAppID = TEXT("None");

UPROPERTY(Config, EditAnywhere, BlueprintReadWrite, Category = "XG")
FString XGAppKey = TEXT("None");
```

- 开发者在 XGAppID/XGAppKey 中填写了第三方服务的密钥
- 打包后，打包产物中的 `DefaultXGSampleSettings.ini` 直接包含明文的 AppID 和 AppKey
- 任何使用解包工具（如 UnrealPak）的用户都能提取并获取密钥

## 应对方案

### 选项一：项目加密（推荐）

在 Project Settings > Packaging 中启用加密选项。加密后的 pak 文件需要对应的解密密钥才能读取。

**特点**：

- 保护整个 pak 内容，不仅限于 INI
- 引擎内置支持，无需额外插件
- 需要妥善管理解密密钥本身的安全性

### 选项二：服务端密钥管理

不在客户端存储私密密钥。客户端发起请求时携带会话 Token，服务端持有真正的密钥进行 API 调用。

**特点**：

- 最安全的方案，客户端完全不持有敏感信息
- 需要额外的后端基础设施支持

### 选项三：密钥混淆/分割

将密钥拆分为多段，在代码中拼接：

```cpp
FString GetApiKey()
{
    FString Part1 = TEXT("a1b2");
    FString Part2 = TEXT("c3d4");
    FString Part3 = TEXT("e5f6");
    return Part1 + Part2 + Part3;
}
```

**特点**：

- 仅增加逆向难度，不能完全阻止
- 配合编译优化可使字符串不直接出现在二进制中
- 属于 治标不治本 的方案

## 核心原则

- **不要将任何密钥硬编码到客户端中**，无论以何种形式
- 如果密钥必须存在于客户端（如推送证书），**必须启用 pak 加密**
- 在代码评审阶段，检查所有 `UPROPERTY(Config)` 属性，确认没有敏感信息外泄
