# 加锁与线程安全

## 概述

多线程环境下，多个线程同时访问同一份数据会导致竞态条件（Race Condition）。UE 提供多种机制来保护共享数据：FCriticalSection、FScopeLock、FThreadSafeBool 以及 C++ 标准库的 std::atomic。

## 竞态条件演示

当两个线程同时读写同一个 `bool` 变量而不加锁时：

```
时间线：
Thread A (Run): 读取 bRunning → true → 继续工作 → ...
Thread B (Stop):                    → 写入 bRunning = false → ...
Thread A (Run):  → 读取 bRunning → ???
```

由于 CPU 指令重排和缓存不一致，Thread A 可能**永远读不到** Thread B 写入的值，导致"已经 Stop 了但线程还在跑"的现象。这就是竞态条件。

## FCriticalSection + FScopeLock

### 基本使用

```cpp
FCriticalSection CriticalSection;

void WriteData()
{
    FScopeLock Lock(&CriticalSection);
    // 受保护的写操作
    bRunning = false;
}

void ReadData()
{
    FScopeLock Lock(&CriticalSection);
    // 受保护的读操作
    if (!bRunning) break;
}
```

`FScopeLock` 是 RAII 类：
- 构造时调用 `CriticalSection.Lock()` 尝试获取锁
- 析构时调用 `CriticalSection.Unlock()` 释放锁
- 离开作用域自动释放，不会忘记解锁

### 在 FRunnable 中的标准模式

```cpp
// XGSimpleRunnable.h
FCriticalSection CriticalSection;
bool bRunning = true;

// XGSimpleRunnable.cpp
uint32 FXGSimpleRunnable::Run()
{
    while (bRunning)  // 首次检查（无锁，快速判断）
    {
        FPlatformProcess::Sleep(0.04f);

        FScopeLock Lock(&CriticalSection);
        if (!bRunning)  // 锁保护下的精确判断
            break;
        
        // 执行工作...
    }
    return 0;
}

void FXGSimpleRunnable::Stop()
{
    // Stop() 在外部线程调用
    FScopeLock Lock(&CriticalSection);
    bRunning = false;
}
```

### 写锁避让策略

上例在 `Run()` 中，每次循环**先进入 Sleep**，然后**加锁检查** `bRunning`。这种"先让出 CPU，再加锁检查"的策略称为**写锁避让**——让写操作（Stop）有更高的机会先执行。

### IsInGameThread() 保护性重分发

另一种常见的线程安全模式：如果某个函数可能被非 GameThread 调用，自动将操作重分派到 GameThread：

```cpp
void UXGThreadSubsystem::SetMyNum(int32 InInt)
{
    if (!IsInGameThread())
    {
        AsyncTask(ENamedThreads::GameThread, [this, InInt]() {
            SetMyNum(InInt);
        });
        return;  // 注意：当前线程直接返回，不执行后续代码
    }

    // 以下代码保证在 GameThread 执行
    MyNum = InInt;
}
```

此模式确保 `MyNum` 的赋值始终在 GameThread 完成。

## FThreadSafeBool

`FThreadSafeBool` 是基于 `FThreadSafeCounter`（底层使用 `std::atomic<int32>`）封装的原子布尔类型：

```cpp
#include "HAL/ThreadSafeBool.h"

FThreadSafeBool bRunning = true;
```

特点：
- 读写操作是**原子的**，不需要额外加锁
- 使用方法与普通 `bool` 相同
- 内部通过 `std::atomic<int32>` 实现

### 使用 FThreadSafeBool 的 FRunnable

```cpp
// XGSimpleRunnableBool.h
FThreadSafeBool bRunning = true;

// XGSimpleRunnableBool.cpp
uint32 FXGSimpleRunnableBool::Run()
{
    while (bRunning)  // 原子读，无需锁
    {
        FPlatformProcess::Sleep(0.04f);
        // 无需 ScopeLock
    }
    return 0;
}

void FXGSimpleRunnableBool::Stop()
{
    // Stop() 在外部线程调用
    bRunning = false;  // 原子写，无需锁
}
```

与 FCriticalSection 版本相比，FThreadSafeBool：
- 代码更简洁（不需要 FCriticalSection 成员和 FScopeLock）
- 性能更高（原子操作轻量于锁）
- 避免锁竞争和死锁风险

### 推荐：使用 std::atomic\<bool\>

UE 官方对 `FThreadSafeBool` 的态度是**已弃用但仍广泛使用**。在可能的情况下，推荐使用 C++ 标准库的 `std::atomic<bool>`：

```cpp
#include <atomic>

std::atomic<bool> bRunning{true};
```

`std::atomic<bool>` 的优势：
- 是 C++ 标准，跨平台和跨引擎可移植
- 语义明确
- 不会被引擎版本更新影响

## 线程不安全的容器

UE 的常见容器**默认不是线程安全的**：

| 容器 | 线程安全 |
|------|----------|
| `TArray` | ❌ 不安全 |
| `TMap` | ❌ 不安全 |
| `TSet` | ❌ 不安全 |
| `FThreadSafeCounter` | ✅ 安全（原子 int32） |
| `FThreadSafeBool` | ✅ 安全（原子 bool） |

多线程环境下访问共享容器时，必须使用 `FCriticalSection` 保护，或使用专门的线程安全容器。

## 线程安全的日志输出

在多线程环境中直接使用 `UE_LOG` 可能产生日志交错或崩溃。推荐使用 AsyncTask 将日志推送到 GameThread：

```cpp
static void PrintWarning(FString InStr)
{
    AsyncTask(ENamedThreads::GameThread, [InStr]() {
        UE_LOG(LogTemp, Warning, TEXT("ThreadLog:[%s]"), *InStr);
    });
}
```

## 线程安全的 Get/Set 方法

Subsystem 中为共享变量提供 Get/Set 接口时，需要在 Set 方法中做线程保护：

```cpp
void UXGThreadSubsystem::SetMyNum(int32 InInt)
{
    if (!IsInGameThread())
    {
        AsyncTask(ENamedThreads::GameThread, [this, InInt]() { SetMyNum(InInt); });
        return;
    }
    MyNum = InInt;
}

int32 UXGThreadSubsystem::GetMyNum()
{
    return MyNum;  // 如果可能被多线程读取，也需要保护
}
```
