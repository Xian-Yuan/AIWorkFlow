# 死亡与重生系统

## 整体架构

死亡系统由三个组件协作完成：

```
HealthComponent（生命管理 + 死亡状态机）
    → 发送 GameplayEvent_Death 事件
        → AbilitySystemComponent 接收事件
            → 激活 XGRPGGameplayAbility_Death（死亡能力）
                → ActivateAbility（取消普通能力，播放死亡动画）
                → EndAbility（调用 FinishDeath，销毁角色）
```

## HealthComponent

`XGRPGHealthComponent` 管理角色的生命值和死亡状态。

### 死亡状态机

```cpp
enum class EDeathState : uint8
{
    NotDead,         // 存活
    DeathStarted,    // 死亡开始（播放死亡动画中）
    DeathFinished    // 死亡完成（执行销毁逻辑）
};
```

状态转换规则：
- `NotDead → DeathStarted`：条件成立时（生命值 ≤ 0）
- `DeathStarted → DeathFinished`：死亡能力执行完毕
- **禁止逆向转换**：一旦进入死亡状态不可回退

### OutOfHealth 事件

```cpp
void UXGRPGHealthComponent::HandleHealthChanged(float NewHealth, float MaxHealth)
{
    if (NewHealth <= 0.0f && GetDeathState() == EDeathState::NotDead)
    {
        SetDeathState(EDeathState::DeathStarted);
        // 服务端发送 GameplayEvent
        if (ASC && GetOwnerRole() == ROLE_Authority)
        {
            FGameplayEventData EventData;
            EventData.Instigator = GetOwner();
            ASC->SendGameplayEventToActor(
                GetOwner(),
                XGRPGGameplayTags::GameplayEvent_Death,
                EventData);
        }
    }
}
```

`FindHealthComponent` 是一个静态辅助方法，用于从任何 Actor 快速获取 HealthComponent。

## 死亡能力（XGRPGGameplayAbility_Death）

### 关键设计

死亡能力通过 **GameplayEvent** 触发，而非传统的 InputTag 手动激活方式。这个设计参考了成熟项目（如 Lyra）的死亡处理模式。

```cpp
UXGRPGGameplayAbility_Death::UXGRPGGameplayAbility_Death(const FObjectInitializer& ObjectInitializer)
    : Super(ObjectInitializer)
{
    InstancingPolicy = EGameplayAbilityInstancingPolicy::InstancedPerActor;
    NetExecutionPolicy = EGameplayAbilityNetExecutionPolicy::ServerInitiated;
    
    if (HasAnyFlags(RF_ClassDefaultObject))
    {
        FAbilityTriggerData TriggerData;
        TriggerData.TriggerTag = XGRPGGameplayTags::GameplayEvent_Death;
        TriggerData.TriggerSource = EGameplayAbilityTriggerSource::GameplayEvent;
        AbilityTriggers.Add(TriggerData);
    }
}
```

### 注册方式

死亡能力的注册方式不同于其他能力——它在角色基类中单独注册，不需要 InputTag：

```cpp
// 在 XGRPGCharacterBase 中单独注册死亡能力
UClass* DeathAbilityClass = DeathAbilityClasses.LoadSynchronous();
if (DeathAbilityClass)
{
    FGameplayAbilitySpec DeathSpec(DeathAbilityClass, 1, INDEX_NONE, this);
    DeathAbilityHandle = ASC->GiveAbility(DeathSpec);
}
```

`INDEX_NONE` 表示没有 InputTag，因为死亡能力不需要通过按键触发。

### 激活与终止

```
ActivateAbility
    → 取消所有不包含 Ability_Behavior_SurvivesDeath 标签的能力
    → 设置 SetCanBeCanceled(false) —— 死亡不可中断
    → 调用 BlueprintImplementableEvent StartDeath()
        → 蓝图中播放死亡蒙太奇
        → 延迟启动布娃娃物理
        → 1.5秒后重生逻辑
        → Wait Delay 5秒 → 调用 EndAbility

EndAbility
    → 检查死亡状态是否为 DeathStarted
    → 调用 BlueprintImplementableEvent FinishDeath()
        → 销毁角色
    → 调父类 Super::EndAbility
```

### CDO 检测

在构造函数中通过 `HasAnyFlags(RF_ClassDefaultObject)` 判断是否为类默认对象。只有 CDO 才注册 TriggerData，避免每个实例重复注册。

```
RF_ClassDefaultObject == true → 注册 TriggerData（仅执行一次）
RF_ClassDefaultObject == false → 跳过 TriggerData 注册（实例化时不重复注册）
```

## 布娃娃物理（Ragdoll）

死亡后播放蒙太奇约 0.2-0.8 秒后启动布娃娃：

```cpp
// 蓝图节点顺序：
1. Set Collision Enabled → Query Only / Physics Only
2. Set Collision Response to Channel → 关闭碰撞
3. Set Simulate Physics → true（开启物理模拟）
4. Add Impulse → 在指定骨骼（如 Pelvis）施加冲击力
```

延迟是因为刚死亡时立即开启布娃娃会中断死亡动画播放，直接"散架"。

## 重生逻辑

重生逻辑写在死亡能力的蓝图 Event Graph 中：

```
Timer（1.5秒后触发）
    → Has Authority（确保只服务端执行）
    → Get Controller
        → Cast to PlayerController（失败则可能是 AI Controller）
    → UnPossess（释放当前控制器）
    → Set Owner（设置为 PlayerController，保持 RPC 通道）
    → Get GameMode → Restart Player（重新生成角色）
```

### 关键坑点

- **AI Controller vs Player Controller**：不能直接将 Controller Cast 为 PlayerController。AI 角色的 Controller 是 AIController，Cast 失败会导致崩溃。必须用 Cast 检查后再处理。
- **Set Owner 的目的**：重生后如果所有权丢失，RPC 可能无法送达
- **UnPossess + Restart Player 模式**：先释放控制器，再从 GameMode 重生

### 时间线

```
0.0s：死亡能力激活 → 播放蒙太奇
0.2~0.8s：启动布娃娃物理
1.5s：重生逻辑执行（服务端 Restart Player）
5.0s：Wait Delay 结束 → EndAbility → FinishDeath → 销毁角色
```

## 客户端同步

死亡流程的客户端同步：

```
服务端：PostGameplayEffectExecute 检查生命值 ≤ 0
    → SendGameplayEventToActor（发送 GameplayEvent_Death）
        → 服务端激活死亡能力（ServerInitiated 执行策略）
            → 能力复制到客户端 → 客户端激活死亡能力
                → 播放死亡蒙太奇 → 布娃娃物理
```

`ServerInitiated` 执行策略确保由服务端发起死亡流程，客户端被动同步。

## 注意事项

- 死亡能力不可取消（`SetCanBeCanceled(false)`），防止其他能力中断死亡流程
- `SurvivesDeath` 标签标记那些不被死亡取消的能力（如有需要）
- 重生后需要重新注册能力和属性
- 布娃娃延迟时间需根据死亡动画长度调整
- `GameplayEvent` 触发方式不依赖输入系统，适合自动触发场景
- 死亡能力在蓝图中实现细节（重生逻辑、蒙太奇选择等）
