# 打包与工程配置

## 启动模块

项目启动时通过 `XGRPG` 模块初始化 GAS 系统，模块定义在 `XGRPG.h/.cpp` 中。

## 服务器与客户端配置

### 专有服务器限制

项目在处理网络功能时特别考虑了专有服务器（Dedicated Server）的限制：

- **动画通知不在 DS 上执行**：所有 AnimNotify 逻辑需通过 `GetNetMode() != NM_DedicatedServer` 排除
- **GEEC 仅在服务端执行**：`WITH_SERVER_CODE` 宏保护伤害计算逻辑
- **骨骼刷新**：专有服务器上需要特殊处理 SkeletalMesh 刷新
- **蒙太奇合成**：DS 上的 Montage Blend Settings 需要特殊配置

### 平台宏

```cpp
// GEEC 在服务端执行的保护宏
WITH_SERVER_CODE
{
    // 仅在服务端执行的伤害计算逻辑
}

// 动画通知跳过 DS
if (GetNetMode() != NM_DedicatedServer)
{
    // 客户端执行的动画或视觉逻辑
}
```

## 关卡配置

### GameMode 设置

需要在游戏模式中配置：

- Default Pawn Class：`XGRPGCharacterBase`（或其蓝图子类）
- HUD Class：`AXGRPGHUD`
- Player Controller Class：`XGRPGPlayerController`（或其蓝图子类）

### PlayerController 配置

`XGRPGPlayerController` 需要设置：

- 启用输入处理（`bShowMouseCursor`、输入映射上下文）
- NumberPopComponent 的挂载
- UI 创建（在 OnPossessed 或延迟后）

## 打包设置

### 编辑器项目设置

1. **Asset Manager 扫描**：在 Project Settings → Asset Manager 中配置 PrimaryAssetTypes 的扫描规则
2. **GameplayCue 路径**：GameplayCue 必须放在特定路径下才能正确打包

### GameplayCue 路径规则

GameplayCue 通知类蓝图必须放置在以下路径才能正确打包：

```
Content/GameplayCueNotifies/
```

放在其他路径可能导致打包后 GameplayCue 不生效。这个限制是由 UE 的资产扫描机制决定的——打包时 Asset Registry 只会扫描特定路径下的 GameplayCue 资产。

### 输入系统配置

增强输入系统（Enhanced Input）配置：

1. 创建 Input Actions（每个 InputTag 对应一个 Action）
2. 创建 Input Mapping Context
3. 在 PlayerController 或 Pawn 中设置 Mapping Context
4. 注意打包后输入配置文件的引用必须有效

### 项目设置参考

以下项目设置为开发中使用的配置，打包时需根据目标平台调整：

| 设置项 | 推荐值 | 说明 |
|-------|--------|------|
| Target Hardware | Desktop / Console | 根据目标平台选择 |
| Packaging Method | Cook & Package | 完整打包流程 |
| Build Configuration | Shipping（发布）/ Development（调试） | 发布时使用 Shipping |
| Network Mode | Client-Server | 多人联机 |

## 结语

本章（第三十六章）是本课程中最庞大、最综合的章节，贯穿了 GAS 框架的全部核心概念，并将其整合到一个可运行、可扩展的 RPG 项目中。项目的代码结构兼顾了参考项目的架构经验和实际教学需求。

项目以 GAS 框架为核心，覆盖了以下完整知识链条：

1. **框架层**：GAS 四件套（ASC、GA、AttributeSet、GE）的完整配置与网络同步
2. **战斗层**：从连击、重攻击到远程追踪弹的完整战斗体系
3. **技能层**：冷却、消耗、冲刺、大招、恢复等多类型技能
4. **反馈层**：伤害数字弹出、GameplayCue 受击反馈
5. **系统层**：背包、装备、AssetManager 等支撑系统
6. **部署层**：网络同步、专有服务器、打包配置

通过本章的学习，可以掌握如何在一个真实的 RPG 项目中组织和运用 GAS 体系，理解各组件间的协作关系，以及如何处理多人网络环境下 GAS 的特有挑战。
