## 通用引擎架构

### 17 层通用引擎分层模型

| 层级 | 名称 | 说明 |
|------|------|------|
| 1 | 硬件层 | CPU、GPU、内存、硬盘 |
| 2 | 驱动层 | 图形驱动（DirectX / Vulkan / Metal）、输入驱动 |
| 3 | 操作系统层 | Windows、Linux、Android、iOS 平台 API |
| 4 | 第三方 SDK | PhysX、Havok、Steamworks、Oculus SDK |
| 5 | 平台独立层 | UE 抽象的平台无关接口，屏蔽 OS 差异 |
| 6 | 核心系统 | 字符串（FString / FName / FText）、容器（TArray / TMap / TSet）、数学库（FVector / FRotator / FMatrix / Transform / FQuat）、日志、配置文件 |
| 7 | 资源管理 | 资源加载/卸载、引用追踪、垃圾回收（GC） |
| 8 | 渲染器 | RHI（Render Hardware Interface）、渲染管线、Draw Call 管理 |
| 9 | 视觉效果 | 后处理、粒子系统（Niagara / Cascade）、材质系统 |
| 10 | 碰撞与物理 | 碰撞检测（FCollisionQueryParams）、物理模拟（PhysicsBody / Constraint）、射线检测 |
| 11 | 人机交互（HID） | 输入系统（Action / Axis Mappings）、游戏手柄/键盘/触控 |
| 12 | 游戏前端 | GameMode、GameState、PlayerState、PlayerController、Pawn、Character |
| 13 | 音频 | 音频引擎（FMOD / Wwise 集成）、3D 音效、音频混合 |
| 14 | 动画 | 动画蓝图（AnimBlueprint）、状态机（StateMachine）、混合空间（BlendSpace）、IK、Montage |
| 15 | 网络 | 复制（Replication）、RPC、客户端预测、权威服务器模型 |
| 16 | 脚本 | 蓝图虚拟机（Blueprint VM）、蓝图的编译和执行 |
| 17 | Gameplay 层 | GAS（GameplayAbilitySystem）、EnhancedInput、GameFeature、ModularGameplay |

### 学习方法论

- 不需要一次性掌握所有层，先建立框架，用哪个学哪个
- 步骤建议：框架性认识 → 玩蓝图 → 学 C++ → UE C++ 流程串联 → 做 demo → 逐步扩展
- 推荐资源：
  - **Games104**（B 站/YouTube）— 工业级游戏引擎架构课程
  - **大招老师的 inside-ue4 / inside-ue5 系列文章** — UE 源码分析

### 蓝图与 C++ 分工

| 方面 | 蓝图 | C++ |
|------|------|-----|
| 适用场景 | 大量重复的简单工作、快速迭代、策划配置 | 底层框架、核心代码、性能敏感逻辑 |
| 示例 | 动画蓝图、AI 行为树、UI 事件绑定 | 引擎模块、Gameplay 框架基类、网络底层 |

## Module 体系

### 四大核心概念

| 概念 | 层级 | 说明 |
|------|------|------|
| **Module** | 最基础 | UE 最小编译单元（代码 + 资源文件），可编译为 .dll（Editor 下）或静态链接 |
| **Plugin** | 包含 Module | 可包含一个或多个 Module，也可依赖其他 Plugin。通过 `.uplugin` 文件描述 |
| **Project** | 包含 Module 和 Plugin | 游戏/应用项目主体，通过 `.uproject` 文件描述 |
| **Program** | 与 Project 同级 | 独立控制台程序，不涉及游戏玩法（如 UBT、UAT） |

```
Project / Program
├── Module(s)
└── Plugin(s)
    └── Module(s)
```

### Module 的实体化

- 每个 Module 目录下有一个 `.Build.cs` 文件
- 通过 `.Build.cs` 定义模块的：依赖关系、编译选项（PCHUsage、bUsePrecompiled、OptimizationFlags 等）
- 每个 Module 需要实现 `IModuleInterface` 接口：

| 方法 | 作用 |
|------|------|
| `StartupModule()` | 模块加载时调用，用于初始化 |
| `ShutdownModule()` | 模块卸载时调用，用于清理 |
| `SupportsDynamicReloading()` | 是否支持热重载（Editor 下动态加载/卸载） |

### Build.cs 依赖管理

```csharp
// 公开依赖 — 会传导给依赖本模块的其他模块
PublicDependencyModuleNames.AddRange(new string[] { "Core", "CoreUObject", "Engine" });

// 私有依赖 — 仅本模块可见，不传导
PrivateDependencyModuleNames.AddRange(new string[] { "Slate", "SlateCore" });
```

- 尽量最小化公开依赖，能用私有依赖就用私有依赖
- 公开依赖会扩散——依赖本模块的模块会自动获得所有公开依赖

### 模块加载阶段

| 阶段 | 说明 |
|------|------|
| `PreDefault` | 最早加载的模块，如 Core、CoreUObject、Engine |
| `Default` | 大多数模块在此阶段加载 |
| `PostDefault` | Core 完全初始化后的模块 |
| `EngineInit` | 引擎初始化阶段的模块 |
| `DoNotAutoLoad` | 不自动加载，需要显式调用 |
| `Max` | 枚举最大值，不实际使用 |

### 使用环境

| 环境 | 说明 |
|------|------|
| **Runtime** | 运行时和 Editor 下都可用 |
| **Editor** | 仅在 Editor 下可用，打包后不可使用 |

### 重要约束

- 项目插件**不能引用项目内容**（GameFeature 插件是例外，可以依赖项目内容）
- 引擎模块：`Engine/Source/`；引擎插件：`Engine/Plugins/`
- 项目插件：`Project/Plugins/`；GameFeature 插件：`Project/Plugins/GameFeatures/`
- **循环依赖**必须避免，解决方案是提取公共模块

## 项目目录架构

### 项目根目录结构

```
ProjectRoot/
├── .uproject              # 项目配置文件（JSON）
├── Config/                # 项目配置（分层覆盖）
│   ├── DefaultEngine.ini
│   ├── DefaultGame.ini
│   ├── DefaultInput.ini
│   └── ... (Default*.ini)
├── Content/               # 资源文件（.uasset / .umap）
├── Source/                # C++ 源码
├── Plugins/               # 项目级插件
├── Intermediate/          # 中间构建产物
├── Saved/                 # 运行时生成文件
├── Binaries/              # 编译产物（.dll / .exe）
└── DerivedDataCache/      # DDC 缓存
```

### 目录删除规则

| 目录 | 是否可删 | 说明 |
|------|---------|------|
| `Intermediate/` | ✅ 可删 | 编译缓存和中间产物，重编即可恢复 |
| `Saved/` | ✅ 可删 | 日志、截图、崩溃 dump 文件 |
| `Binaries/` | ✅ 可删 | 编译好的二进制文件，重编即可恢复 |
| `DerivedDataCache/` | ✅ 可删 | DDC 缓存，重新构建即可恢复 |
| `VS/` (Visual Studio) | ✅ 可删 | VS 解决方案文件和缓存 |
| `Source/` | ❌ 不可删 | 源码是核心资产 |
| `Config/` | ❌ 不可删 | 项目配置 |
| `Content/` | ❌ 不可删 | 资源文件 |
| `.uproject` | ❌ 不可删 | 项目入口文件 |
| `Build/` | ⚠️ 视情况 | 通常可删，部分项目（如 Lyra）需要保留 |

**崩溃 dump 定位**：`Saved/Crashes/` 目录下

### .uproject 文件结构（JSON）

```json
{
    "FileVersion": 3,
    "EngineAssociation": "5.3",
    "Modules": [
        {
            "Name": "MyProject",
            "Type": "Runtime",
            "LoadingPhase": "Default"
        }
    ],
    "Plugins": [
        {
            "Name": "MyPlugin",
            "Enabled": true
        }
    ]
}
```

### Config 分层覆盖机制

```
引擎配置（Base*.ini）← 项目配置（Default*.ini）← 平台配置（Platform*.ini）← 打包后配置
```

- 每层对上层进行覆盖/扩展
- 通过继承 `UDeveloperSetting` 类可在 Config 中注册自定义配置

### Git 版本管理配置

**.gitignore 规范要点：**
- 使用相对路径层级检索模式
- 必须忽略：`Intermediate/`、`Saved/`、`Binaries/`、`DerivedDataCache/`、`.vs/`、`*.opensdf`、`*.sdf`、`*.suo`
- 若使用 Git LFS，还需额外忽略 LFS 追踪文件的两个标记文件
- `Source/`、`Config/`、`Content/`、`Plugins/`、`.uproject` 必须纳入版本管理

**.gitattributes（Git LFS）配置示例：**

```
*.uasset filter=lfs diff=lfs merge=lfs -text
*.umap filter=lfs diff=lfs merge=lfs -text
*.pak filter=lfs diff=lfs merge=lfs -text
*.dll filter=lfs diff=lfs merge=lfs -text
*.exe filter=lfs diff=lfs merge=lfs -text
*.pdb filter=lfs diff=lfs merge=lfs -text
```

被 LFS 追踪的文件在仓库中存储的是指针文件，而非实际二进制内容。

### 插件目录结构

**标准写法：**
```
PluginName/
├── Source/
│   ├── PluginName/
│   │   ├── Public/
│   │   │   └── *.h
│   │   └── Private/
│   │       └── *.cpp
│   └── PluginName.Build.cs
├── Resources/
│   └── Icon128.png           # 插件图标（128x128 PNG）
└── PluginName.uplugin
```

**第三方库写法：**
```
PluginName/
├── Source/
│   ├── PluginName/
│   ├── PluginName.Build.cs
│   └── ThirdParty/
│       └── LibraryName/
│           ├── lib/
│           ├── include/
│           └── ...
├── Resources/
├── PluginName.uplugin
└── README.md
```

**Game Feature 插件** — 特殊类型：
- 放在 `Project/Plugins/GameFeatures/` 目录下
- **可以依赖项目内容**（唯一的例外）
- 有独立生命周期管理和资产加载机制
- 适合 DLC、赛季内容等模块化功能

### UHT 生成文件位置

`Intermediate/Build/Win64/Editor/Inc/项目名/UHT/*.generated.h`

> `.generated.h` 文件必须手动 `#include` 到对应的头文件末尾

## 引擎目录架构

### 引擎根目录结构

```
Engine/
├── Binaries/       # 可执行文件和预编译程序
├── Build/          # 构建脚本和平台配置
├── Config/         # 引擎级配置（Base*.ini）
├── Content/        # 引擎内置资源（含世界分区必需文件）
├── Plugins/        # 引擎插件（含 MarketPlace 下载的）
├── Programs/       # 独立程序
├── Saved/          # 日志、崩溃报告、自动保存
├── Shaders/        # 着色器源码（.ush / .usf，HLSL 语法）
└── Source/         # 引擎源码
    ├── Runtime/    # 运行时模块
    ├── Developer/  # 开发工具
    ├── Editor/     # 编辑器模块
    └── Programs/   # 独立程序源码
```

### 关键目录详解

| 目录 | 是否可删 | 说明 |
|------|---------|------|
| `Engine/Binaries/` | ❌ 不可删 | 含预编译程序，编辑器启动需要；只有源码版引擎重编才会重新生成 |
| `Engine/Build/` | ❌ 不可删 | 构建脚本和平台配置 |
| `Engine/Config/` | ❌ 不可删 | 引擎级 Base*.ini 配置 |
| `Engine/Content/` | ❌ 不可删 | 含世界分区必需文件，不能 .gitignore |
| `Engine/Plugins/MarketPlace/` | ⚠️ | 商城下载的插件放这里 |
| `Engine/Saved/` | ✅ 可删 | 日志和崩溃报告 |
| `Engine/Shaders/` | ❌ 不可删 | 着色器源码 |

### Engine/Source/ 四大模块分类

| 分类 | 说明 | 关键模块示例 |
|------|------|-------------|
| **Runtime/** | 运行时必需 | Core、UObject、Engine、RenderCore、OnlineSubsystem、Slate |
| **Developer/** | 开发工具 | AssetRegistry、MessageLog |
| **Editor/** | 编辑器特有 | UnrealEd、LevelEditor、BlueprintEditor、AnimationEditor、MaterialEditor |
| **Programs/** | 独立程序源码 | |

### Programs/ 中的关键独立程序

- **UBT**（Unreal Build Tool）— 构建工具，基于 `.Build.cs` 和 `.Target.cs` 定义编译规则
- **UAT**（Unreal Automation Tool）— 自动化打包部署工具
- **BlankProgram** — 编写独立控制台程序的模板

### 路径引用注意事项

引擎 Content 目录下统一使用 `/Game/` 路径前缀，在制作弱引用时路径替换需要特别注意。

## VS 编译类型

### 烘焙（Cooking）概念

- 编辑器下的资产不能直接在目标平台上使用，需要**烘焙**转化为目标平台可用的格式
- 烘焙前后的资产**不能混用**：
  - 不能在编辑器加载烘焙后的资产
  - 不能在打包后加载编辑器原生资产
- 不同引擎版本打包后的 `.pak` 文件不能互相使用

### UBT / UHT 工作流

1. 读取 `.uproject` → 获取项目信息和关联的引擎版本
2. 读取 `Source/` 下的 `.Target.cs` → 确定目标类型
3. 递归解析所有模块的 `.Build.cs` → 确定依赖树
4. 调用 **UHT**（Unreal Header Tool）→ 解析 UE 宏（`UPROPERTY` / `UFUNCTION` / `UCLASS` / `USTRUCT` / `UENUM` / `UINTERFACE` / `UDELEGATE`），生成反射代码（`.generated.h`）
5. 将编译配置传递给 MSBuild / Clang 执行编译

### 编译配置体系（5 级 × 4 目标）

**配置级别**（由低到高）：`Debug` → `DebugGame` → `Development` → `Test` → `Shipping`

| 配置 | 调试能力 | 优化 | 日志 | 使用场景 |
|------|---------|------|------|---------|
| **Debug** | 引擎+游戏全可调试 | 无优化 | 有 | 深入调试引擎代码 |
| **DebugGame** | 仅游戏代码可调试，引擎用 Development | 引擎有优化 | 有 | **推荐开发配置** |
| **Development** | 仅编辑器调试 | 部分优化 | 有 | Editor 常规运行 |
| **Test** | 有限调试 | 大部分优化 | 有 | 性能测试 |
| **Shipping** | 无调试符号 | 全部优化 | 无控制台日志 | **最终发布版本** |

**编译目标：**

| 目标 | 说明 |
|------|------|
| （空）= Game | 独立游戏程序 |
| Editor | 编辑器 |
| Client | 纯客户端（需 Target.cs 配置） |
| Server | 专用服务器（需 Target.cs 配置） |

### Client / Server 区分

- 通过 `.Target.cs` 中的 `TargetType` 定义
- UE 使用同一套代码开发客户端和服务器，通过 `IsServer()` / `IsClient()` / `HasAuthority()` 等判断
- **不推荐**在 VS 直接启动 Server 类型（跑不起来），应在编辑器中以 **Play As Client** 或 **Listen Server** 方式启动

### 代码优化与调试冲突

**现象**：Development Editor 下断点乱跳/无法命中（编译器优化了代码）

**解决方案：**
1. 切换到更详细的编译配置（如 `DebugGame Editor`）
2. 使用**防止优化宏**包裹关键函数：
   - Windows 原生：`#pragma optimize("", off/on)`
   - UE 封装：`UE_DISABLE_OPTIMIZATION` / `PRAGMA_DISABLE_OPTIMIZATION`
3. 在 `.Build.cs` 中对该模块关闭优化

### 不推荐的操作

- ❌ 依赖 VS 的 **Error List** 窗口看 UE 编译错误——应直接看**输出窗口（Output Window）**
- ❌ 在 VS 中直接启动 Server 目标——应使用编辑器的 Play 功能
