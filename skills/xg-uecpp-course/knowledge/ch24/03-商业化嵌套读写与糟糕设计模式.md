# 商业化嵌套读写与糟糕设计模式

## 概述

本章展示两种实际商业开发中会遇到的特殊 JSON 设计模式：BadMessage（网关嵌套 JSON 字符串）和 VeryBadJson（动态 Key 名）。前者在大公司分布式架构中常见且有合理原因；后者是应避免的反模式。

## BadMessage — Info 字段为 JSON 字符串的嵌套模式

### 场景来源

分布式服务器架构中，外层一级网关统一报文格式，内层二级业务服务各不同：

```
一级网关（统一入口）
  └── Code / Message / Info / Sid（外层固定字段）
       └── Info（String 类型，内容是另一个 JSON 字符串）
            └── 二级业务字段（Code2 / Message2 / Data...）
```

这种架构的好处是**一级接口格式永远不变**，二级服务各自独立演化不影响外层。

### 序列化（写入）

```cpp
FBadMessageInfo MyBadInfo;
MyBadInfo.Code = 0;
MyBadInfo.Message = TEXT("一级服务接口");
MyBadInfo.Sid = TEXT("123456");

// 构造二级服务数据并转为 JSON 字符串
TempMessageInfo MyInfo;
MyInfo.UECode = 564654;
MyInfo.UEMessage = TEXT("二级服务接口");
MyBadInfo.Info = MyInfo.ToString();  // Info 是 JSON 字符串，不是 Object！

// 外层序列化
TSharedRef<TJsonWriter<...>> Writer = TJsonWriterFactory<...>::Create(&JsonStr);
Writer->WriteObjectStart();
Writer->WriteValue(TEXT("Code"), MyBadInfo.Code);
Writer->WriteValue(TEXT("Message"), MyBadInfo.Message);
Writer->WriteValue(TEXT("Info"), MyBadInfo.Info);    // 写的是字符串！
Writer->WriteValue(TEXT("Sid"), MyBadInfo.Sid);
Writer->WriteObjectEnd();
Writer->Close();
```

**关键点**：`Info` 字段类型是 `FString`（字符串），不是 `FJsonObject`（对象）。它存储的是另一个"已序列化的 JSON 字符串"。

### 反序列化（读取）

```cpp
// 读外层
FBadMessageInfo BackInfo;
BackInfo.Code = JsonObject->GetNumberField(TEXT("Code"));
BackInfo.Message = JsonObject->GetStringField(TEXT("Message"));
BackInfo.Info = JsonObject->GetStringField(TEXT("Info"));    // 拿到 JSON 字符串
BackInfo.Sid = JsonObject->GetStringField(TEXT("Sid"));

// 解析内层：将 Info 字符串再次反序列化为结构体
TempMessageInfo MyBackInfo = TempMessageInfo::FromString(BackInfo.Info);
```

**二次反序列化**：先用 `GetStringField` 取出 Info（字符串），再调用内层结构体的 `FromString` 将字符串转为结构体。

### 数据结构定义

```cpp
struct FBadMessageInfo
{
    int32 Code = -1;
    FString Message;    // 一级服务描述
    FString Sid;
    FString Info;       // 二级服务的完整 JSON 字符串（非 Object！）
};
```

## VeryBadJson — 动态 Key 名的糟糕设计

### 问题描述

后端将数据以 **Key 名本身** 承载信息，而非用数组 + 固定字段：

```json
// ❌ 糟糕设计：Key 名携带业务数据
{
  "Player_XG_abc123": 5.6,
  "Player_CC_def456": 3.7,
  "Player_BB_ghi789": 15.6
}

// ✅ 正确设计：数组 + 固定结构
{
  "Players": [
    {"Name": "XG", "Id": "abc123", "HP": 5.6},
    {"Name": "CC", "Id": "def456", "HP": 3.7}
  ]
}
```

### 为什么这是糟糕设计

- Key 名每次不同（GUID），无法用 `GetStringField("固定key")` 读取
- `FJsonObjectConverter` 自动反射序列化完全不可用
- 只能用 `FJsonObject::Values`（TMap）遍历所有键值对
- 嵌套在其他 JSON 中时解析极其痛苦

### 序列化（写入 VeryBadJson）

```cpp
TSharedRef<TJsonWriter<...>> Writer = TJsonWriterFactory<...>::Create(&JsonStr);
Writer->WriteObjectStart();
for (int i = 0; i < 4; i++)
{
    FString PlayerName = FGuid::NewGuid().ToString();  // 动态生成 Key 名
    Writer->WriteValue(PlayerName, i);                  // Key = GUID, Value = 索引
}
Writer->WriteObjectEnd();
Writer->Close();
```

### 反序列化（解析动态 Key）

```cpp
TSharedPtr<FJsonObject> JsonObj;
FJsonSerializer::Deserialize(Reader, JsonObj);

TMap<FString, float> PlayerInfos;
for (auto& Pair : JsonObj->Values)   // Values 是 TMap<FString, TSharedPtr<FJsonValue>>
{
    PlayerInfos.Add(Pair.Key, Pair.Value->AsNumber());
}
```

`FJsonObject::Values` 是 `TMap<FString, TSharedPtr<FJsonValue>>`，遍历时：
- `Pair.Key` = 动态 Key 名
- `Pair.Value->AsNumber()` = 对应的值

### 判断与对策

| 模式 | 出现场景 | 应对 |
|------|----------|------|
| BadMessage（JSON 字符串嵌套） | 分布式网关架构，合理 | 二次反序列化 |
| VeryBadJson（动态 Key 名） | 糟糕的后端设计 | 尽量在设计阶段拒绝，否则用 Values 遍历 |
| NotVeryBadJson（固定结构数组） | 标准做法 | 首选此方案 |
