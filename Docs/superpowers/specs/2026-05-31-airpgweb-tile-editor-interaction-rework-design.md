# AIRPGWeb Tile Editor Interaction Rework Design

日期：2026-05-31
项目：`g:\UEGameDevelopment\Project\AIRPGWeb`
状态：设计已确认，待审阅

## 1. 目标

本设计用于修正当前地图编辑器在“部分地块无法绘制、拖拽无实时反馈、工具分组混乱、预览与实际不一致、模块边界混杂”上的问题。

本轮目标不是继续叠加新工具，而是把编辑器交互模型收敛成稳定、可理解、可扩展的结构。

本轮完成后应满足：

- 开口层地块可以稳定绘制
- 结构层 `outer-wall-h / outer-wall-v` 可以稳定绘制
- 调色板预览、实时预览、最终落格三者表现一致
- 长按拖拽时实时看到将要落下的 tile
- 工具和设置分组清晰
- 当前模块只承担“地图绘制”职责，不再承接运行时地图设置职责

## 2. 问题归类

### 2.1 当前已知问题

- 部分地块可选中但无法绘制
- 开口层全部地块绘制异常
- 结构层 `外墙横 / 外墙竖` 绘制异常
- 拖拽绘制只能在松手后看到结果
- `fill / rectangle-outline / rectangle-fill` 与 `brush` 平级，用户心智不自然
- `设为运行时地图` 混在“地图绘制”模块里，职责边界不清晰
- 部分结构 tile 在调色板 UI 中的显示与最终实际落格内容不一致

### 2.2 问题本质

这些问题并不是彼此独立的散点缺陷，而是共同指向以下三个根因：

1. 工具状态机模型过于扁平
2. 预览系统与最终渲染系统没有统一真相源
3. 当前模块承担了不属于“地图绘制”的职责

因此本轮不建议只补局部 bug，而是以交互模型收敛为主，再顺带消除局部异常。

## 3. 范围

### 3.1 本轮包含

- 工具系统重构
- 画布实时预览层
- 调色板预览统一
- 绘制状态可观测提示
- 顶部工具区分组
- “地图绘制”与“地图编辑”模块边界预留
- 当前页面内“世界拼接”相关 UI 入口移出
- 网页内 AI 配置入口与房屋分步绘制流程
- 相关测试更新

### 3.2 本轮不包含

- 世界布局编辑功能正式实现
- 运行时地图发布迁移后的完整新模块
- 村落或更大范围地图的 AI 自动规划流程
- 复杂多选区
- 自由变换
- 旋转/镜像
- 高级图章刷

## 4. 设计原则

### 4.1 工具分层优先于按钮堆叠

用户首先选择“做什么”，再选择“如何做”。

因此工具分为：

- 顶层工具
- 工具子模式

而不是把所有操作都直接铺成并列按钮。

### 4.2 预览与结果必须同源

调色板、拖拽预览、最终落格都必须来自同一份 tile skin 真相源，避免出现“选的是 A，画出来像 B”的感知错误。

### 4.3 实时反馈优先

拖拽绘制时用户必须实时看到将要产生的结果，不能等到松手后才知道有没有画上。

### 4.4 绘图模块保持单一职责

“地图绘制”模块只负责：

- 选层
- 选 tile
- 绘制
- 清除
- 保存草稿

与运行时地图发布、世界模块布局相关的行为应从当前模块移出。

### 4.5 AI 必须受流程约束

AI 在当前编辑器中的角色不是“一次性自由生成整张地图”，而是受约束的分步绘制助手。

第一阶段必须满足：

- 只处理“职业生活居所 / 房屋”这一类单体建筑需求
- 每一步先生成预览，再由用户确认
- 每一步只解决一个子目标，不跨步混写
- 生成结果必须能映射到当前编辑器已有的图层、tile 与工具

这样可以避免：

- AI 直接输出当前项目不支持的内容
- 一步生成整套布局导致错误难以回退
- 房屋流程与未来村落流程混在一起

## 5. 工具系统重构

### 5.1 顶层工具

顶层工具保留为：

- `brush`
- `eraser`
- `picker`
- `undo`
- `redo`

### 5.2 画笔子模式

`brush` 下的子模式收敛为：

- `brush`
- `fill`
- `rectangle-outline`
- `rectangle-fill`

行为说明：

- `brush`：多功能笔刷。单击时绘制单格，长按拖动时自动连续拖刷
- `fill`：单击连通区域填充
- `rectangle-outline`：拖出边框矩形
- `rectangle-fill`：按下起点并拖到终点，实时预览并铺满整个矩形区域

本轮确认不再把以下行为单独暴露为独立子工具：

- `stroke`

原因：

- `stroke` 已被 `brush` 自然覆盖，不需要再额外占一个入口
- `rectangle-fill` 已被用户确认为高频绘制能力，应作为常驻子工具保留

切换规则：

- 左侧顶层仍只显示 `画笔`
- 点击 `画笔` 后，在其右侧紧贴弹出子工具面板
- 用户选择 `fill`、`rectangle-outline` 或 `rectangle-fill` 后，当前主工具状态切换为对应子工具
- 状态栏与工具高亮都应反映当前激活子工具，而不是只显示笼统的“画笔”

### 5.3 画笔弹出方式

画笔子工具不放在顶部横向工具栏内长期展开，而采用类似 Photoshop 的轻量弹出面板。

但该面板不应作为左侧工具栏内部的普通子节点处理，而应实现为：

- 全局覆层 UI
- 通过画笔按钮坐标进行锚定定位
- 显示位置仍然紧贴在左侧 `画笔` 按钮右侧

推荐实现方式：

- 使用 portal 或统一 overlay layer 渲染
- 打开时计算画笔按钮的屏幕坐标
- 以该坐标为锚点摆放子工具面板

这样可以避免：

- 被左侧栏容器裁切
- 被画布容器或滚动区域遮挡
- 在不同浏览器/内嵌预览环境中出现层级不一致

并保证：

- 打开后紧贴 `画笔`
- 不漂浮到右侧大栏位
- 只展示当前工具的少量子项，不与图层/地块栏混排

这样可以保证：

- 左侧主工具栏保持稳定
- 子工具切换路径短
- 右侧空间继续优先留给图层与当前层地块
- 画笔子工具不会因容器层级问题被遮挡

### 5.4 橡皮工具

`eraser` 默认模式改为：

- `rectangle-fill erase`

即：

- 鼠标按下记录锚点
- 拖动形成矩形区域
- 实时预览待清除区域
- 松手后把目标格写为 `null`

本轮不强制实现橡皮的其他子模式，但后续可扩展：

- `single erase`
- `stroke erase`
- `fill erase`

### 5.5 吸管工具

`picker` 行为保持单击读取当前层 tileId。

读取完成后：

- 更新当前选中 tile
- 保持当前顶层工具不自动跳走

这样用户可以先吸管取样，再继续画笔操作。

## 6. 实时预览层

### 6.1 双层显示模型

画布表现拆成两层：

- `committed layer`：已经写入地图状态的数据
- `preview layer`：当前拖拽手势正在预览、尚未提交的数据

### 6.2 预览行为

#### 画笔拖刷

- 按下开始记录轨迹
- 移动时实时计算将要落下的连续格集合
- 在 `preview layer` 中即时显示真实 tile skin
- 松手后一次性写入 reducer

#### 边框矩形

- 按下记录锚点
- 移动时生成当前矩形边框格集合
- 预览层显示真实 tile 结果，而不是只有虚线框
- 虚线框可作为辅助，但不能替代 tile 预览

#### 填充矩形

`rectangle-fill` 作为当前首批 UI 暴露的画笔子工具。

行为约束：

- 行为同矩形工具
- 预览层显示完整铺满结果
- 入口放在 `画笔` 子工具面板中，与 `边框矩形` 并列
- 松手后一次性把矩形区域内全部格子提交到 reducer
- 若当前 tile 或 layer guard 不允许批量写入，应阻止提交并给出状态提示

#### 橡皮矩形

- 预览层使用半透明“清除蒙层”表示将要被擦除的范围

### 6.3 提交规则

- `pointerup` 后提交 `preview layer` 对应的坐标集合
- 提交成功后清空预览
- 若操作被 guard 阻止，则不生成提交结果，并给出状态提示

## 7. 预览与皮肤统一

### 7.1 真相源

以下三个入口必须复用同一套视觉渲染路径：

- 调色板预览
- 画布实时预览
- 画布最终落格

推荐统一走：

- `tile-topology.ts`
- `TileSkin.tsx`
- `App.css`

### 7.2 调色板改造

当前调色板的字符型 `preview` 只保留为回退文本，不再作为结构 tile 的主预览方式。

结构层与开口层 tile 在调色板中应直接展示真实 skin 缩略图。

这样可以解决：

- `外墙左上角` 在按钮里显示一种形态，落到地图上又是另一种形态
- 用户选错 tile family 但无法即时感知

## 8. 绘制异常的可观测性

### 8.1 状态栏最小提示

状态栏至少应能输出：

- 当前层
- 当前顶层工具
- 当前子模式
- 当前选中 tile
- 当前操作是否被阻止

例如：

- `当前层：开口层，工具：画笔/拖刷，tile：横门开`
- `当前 tile 不适合边框矩形`
- `结构层已锁定，无法修改`

### 8.2 内部状态排查点

实现层应便于区分以下失败位置：

- palette 选中失败
- preview 未生成
- preview 已生成但 reducer 未提交
- reducer 已提交但 skin 未渲染
- 实际写入的是别的 layer

这部分不必暴露成复杂 UI，但设计上要预留独立状态节点，避免以后再次出现“选中了但无法绘制”时只能盲查。

## 9. 顶部信息栏与移出项

### 9.1 顶部信息栏

顶部区域从“大型工具按钮区”收缩为紧凑信息栏，仅保留：

- 小标题
- 地图名称输入
- 地图尺寸
- 当前层
- 当前主工具/子工具
- 保存
- 另存为

工具主入口不再放在顶部，而迁移到左侧竖向主工具栏。

### 9.2 暂时移出当前模块

从“地图绘制”模块移出：

- `设为运行时地图`
- 与世界模块放置相关的入口
- `退出开发者模式`

原因：

- 这些行为属于地图发布/运行时配置或全局导航，不属于地图绘图动作
- 放在当前页面会稀释主任务并增加状态混乱

补充边界：

- 本轮只移出当前页面 UI，不删除相关底层文件与逻辑
- 后续若恢复世界拼接能力，应进入独立模块或独立面板，不回到当前绘图主界面

## 10. AI 辅助绘制流程

### 10.1 第一阶段目标

AI 的第一阶段目标不是“生成整个村落”，而是辅助生成单体房屋或生活居所。

典型输入示例：

- `给我一个铁匠的生活居所`
- `做一个猎人的木屋`
- `设计一个草药师的小型住宅`

第一阶段只覆盖：

- 职业/用途解析
- 房屋骨架生成
- 地板铺设
- 家具摆放

明确不覆盖：

- 村落规划
- 多建筑联动
- 大地图分区

### 10.2 交互方式

AI 必须采用“逐步确认”模式：

1. 用户输入房屋需求
2. AI 只生成当前步骤的预览结果
3. 用户确认后，当前步骤才真正写入画布
4. 系统再进入下一步

不允许第一阶段默认采用：

- 一次性整屋自动绘制
- 跳过用户确认直接连写多步

### 10.3 阶段拆分

房屋流程按以下顺序推进：

1. `structure`：房屋骨架
2. `floor`：地板铺设
3. `furniture`：家具摆放

各阶段职责：

- `structure`
  - 生成外墙轮廓
  - 生成门的基础位置
  - 必要时生成简单内部隔断
- `floor`
  - 根据房间区域铺设地板
  - 不修改已确认的墙体和门位
- `furniture`
  - 在不堵门、不堵主通道的前提下给出家具摆放建议
  - 第一版优先采用规则化点位/槽位摆放

### 10.4 AI 输出约束

AI 不应只返回自由文本，而应返回可被编辑器消费的结构化结果。

推荐输出结构：

- `workflow`: 固定为 `house-residence`
- `step`: 当前阶段
- `summary`: 该步骤的人类可读说明
- `commands`: 当前步骤的绘制命令列表

命令应优先复用当前编辑器已有能力：

- `brush`
- `fill`
- `rectangle-outline`
- `rectangle-fill`

这样可以保证：

- AI 输出与现有手工工具同源
- 每一步都可预览、确认、撤销
- 后续新增村落流程时仍可复用同一命令层

### 10.5 Provider 配置

网页内允许配置 AI Provider，以便接入如智普等第三方模型。

第一阶段建议配置项：

- Provider 名称
- Base URL
- API Key
- Model

实现边界：

- 首版允许本地开发场景下由用户在网页内填写配置
- 但该方式存在密钥暴露风险，不作为长期正式方案
- 更长期的正式方案应由本地或服务端中转保存密钥

## 11. 布局与空间分配

### 11.1 总体布局

地图绘制页面应调整为三段式空间模型：

- 左侧：固定竖向主工具栏
- 中间：最大化画布区域
- 右侧：不再使用普通文档流侧边栏，而改为画布上的覆层面板

该结构的优先级为：

1. 保障中间画布面积最大
2. 保障主工具位置稳定
3. 保障右侧信息栏在不同视口下不掉到底部

### 11.2 左侧主工具栏

左侧仅保留：

- 返回
- 画笔
- 橡皮
- 吸管
- 撤销
- 重做

要求：

- 竖向排布
- 位置固定
- 图标/短文案均可，但不堆叠大段文本
- `退出开发者模式` 从该页面移除

### 11.3 中间画布

中间画布区域是页面最大区域。

要求：

- 页面本身保留纵向滚动能力
- 画布容器内部允许独立滚动
- 顶部标题缩小，只保留必要状态信息
- 地图名、尺寸、当前层、当前工具可合并成紧凑状态条

### 11.4 右侧覆层面板

`图层` 与 `当前层地块` 不再作为普通右侧栏参与整体页面响应式布局，而改为：

- 固定贴在画布右上角
- 作为覆层 UI 压在画布上方
- 视觉上半透明或带模糊背景

原因：

- 在 Trae 内嵌浏览器中，普通右侧栏容易因视口宽度和页面缩放被挤到底部
- 覆层面板不参与外层换行，布局稳定性更高

### 11.5 覆层折叠

`图层` 与 `当前层地块` 两个覆层面板都支持折叠：

- 展开时显示完整内容
- 折叠后保留窄标题条
- 两个折叠条继续竖向贴边排布

目标体验参考 Blender 右侧栏：

- 信息密度高
- 不遮挡过多画布
- 需要时迅速展开

### 11.6 Trae 视口兼容策略

设计上需要显式兼容两个运行环境：

- 独立浏览器（如 Edge）
- Trae 内嵌预览

兼容原则：

- 不依赖外层页面在窄视口下维持三栏文档流
- 关键侧边信息优先改为画布覆层
- 关键弹出层优先锚定到触发按钮附近

这样可以减少：

- 右栏被压到底部
- 用户在 Trae 和独立浏览器里看到两个不同布局
- 工具与状态区被过度拉伸或换行

## 12. 模块边界

### 12.1 当前模块保留

当前模块名称仍为：

- `地图绘制`

其职责限定为：

- 绘图交互
- 图层操作
- tile 选择
- 地图草稿保存

### 12.2 新模块预留

后续新增模块：

- `地图编辑`

未来承接：

- 运行时地图设置

- 模块放置

## 13. 组件与状态结构调整

### 13.1 `editor-reducer.ts`

需要从“扁平工具枚举”升级为：

- 顶层工具状态
- 工具子模式状态
- 预览状态
- 预览提交动作

建议新增或重构的状态包括：

- `activeTool`
- `brushMode`
- `eraserMode`
- `previewCells`
- `previewKind`
- `statusMessage`

并补充：

- `activeBrushSubtool`
- `isBrushPopupOpen`
- `overlayPanelState`

其中 `activeBrushSubtool` 需扩展为：

- `brush`
- `fill`
- `rectangle-outline`
- `rectangle-fill`

并建议补充与 AI 步进确认相关状态：

- `aiWorkflowState`
- `aiPendingStep`
- `aiStepPreview`
- `aiProviderConfig`

### 13.2 `TileCanvas.tsx`

需要承担：

- 手势识别
- 实时预览生成
- 预览层渲染
- 提交触发
- 右侧覆层 UI 的锚点容器

但不承担：

- tile topology 业务判断真相源
- 复杂字符串预览拼装

### 13.3 `TilePalette.tsx`

需要从字符预览列表升级为：

- tile skin 预览列表
- 选中态清晰反馈
- 必要时显示 fallback label

### 13.4 `EditorToolbar.tsx`

需要收缩为紧凑顶部信息栏：

- 小标题
- 地图名称输入
- 当前尺寸/当前层/当前工具状态
- 保存/另存为

## 16. 风险与边界

### 16.1 风险

- 如果只改 UI 分组，不先改状态机，问题仍会反复出现
- 如果只加虚线框而不加真实 tile 预览，拖拽“不可见”问题仍未真正解决
- 如果调色板不复用真实 `TileSkin`，结构 tile 的“看起来和画出来不一致”还会持续存在
- 如果右侧信息栏继续留在普通文档流中，Trae 与独立浏览器之间仍可能出现布局不一致
- 如果画笔子模式仍散落在顶部或右栏，切换路径会变长且不符合用户直觉
- 如果 AI 不按步骤确认而是直接整屋落图，错误会在后续步骤被放大
- 如果 AI 输出不受命令层约束，就会回到“文本建议很多、绘图无从落地”的问题
- 如果前端直接长期保存第三方 API Key，会带来安全风险

### 16.2 边界

本轮不追求高级编辑器，而是把当前编辑器从“能用但容易误解”推进到“稳定、直观、可继续扩展”。

## 17. 确认结论

截至当前，本设计已确认以下要求：

- 部分 tile 无法绘制问题需要纳入正式修复范围
- 拖拽绘制必须实时可见
- `brush` 子工具收敛为 `笔刷 / 填充 / 边框矩形 / 填充矩形`
- `笔刷` 同时承担单击单格与长按拖刷，不再拆出独立 `stroke`
- `eraser` 默认是矩形清除
- 画笔子工具应紧贴画笔按钮右侧弹出
- `图层` 与 `当前层地块` 采用画布右上角覆层 UI
- 覆层面板支持折叠，折叠后继续竖向贴边排布
- 顶部改为紧凑信息栏，不再承担主工具入口
- `设为运行时地图` 从当前 `地图绘制` 模块移出
- 世界拼接只移出当前页面 UI，不删除底层文件
- 后续新增 `地图编辑` 模块承接运行时地图配置与模块放置
- 结构层角块等 tile 的调色板预览与最终落格必须一致
- 设计需兼容独立浏览器与 Trae 内嵌预览两种视口环境
- AI 首个正式流程为“职业生活居所 / 房屋”分步绘制
- AI 必须按“骨架 -> 地板 -> 家具”逐步生成，并在每一步等待用户确认
