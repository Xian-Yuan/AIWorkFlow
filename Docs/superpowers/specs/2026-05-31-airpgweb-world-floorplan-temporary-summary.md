## 1. 文档目的

本文件用于暂时整理当前关于 `AIRPGWeb` 世界中间显示、文本地图、房屋户型图、分层渲染、碰撞与交互表现的阶段性共识。

它的作用是：

- 记录当前已经确认的方向
- 记录已经发现的问题与反例
- 记录下一步继续看“房屋效果”时的设计基准
- 避免在后续迭代中反复回到已经被否定的方案

本文件不是最终设计文档，后续仍会根据预览反馈继续修改。

## 2. 当前总目标

当前优先级不是直接实现整张村庄地图，而是先把以下核心问题设计正确：

1. 中间世界显示不能像小地图缩略图
2. 房屋和村庄布局要有真实逻辑，不能随机堆叠
3. 墙、门、房间、家具、人物、名字必须分层，否则结构必然被挤坏
4. 先看“房屋效果”是否成立，再扩到整村布局

## 3. 已确认方向

### 3.1 世界表现方向

- 采用纯文本/符号表现，不走像素画贴图方案
- 中间区域应更接近“局部场景窗”，不是全村缩略俯视图
- 环境尽量少文字说明，多用符号、线框、道具和空间关系表达
- NPC 名字保留，用于对比尺寸和空间比例
- 玩家名字使用不同颜色，当前倾向绿色

### 3.2 村庄组织方向

- 村庄应有中心逻辑，不再随机摆放建筑
- 村里应有一个中心广场
- 广场中央有水井
- 主路从广场延展出去
- 支路连接到院门或屋门，而不是这里一段那里一段
- 广场不需要木牌
- 住宅或作坊门口可以有木牌，用于点击或悬停显示建筑名称

### 3.3 房屋表现方向

- 房屋必须先按户型图逻辑设计，再生成场景表现
- 墙和门的连通方式应参考真实户型图/施工图
- 房屋整体应形成封闭空间
- 内部房间之间应有明确分隔
- 门必须嵌在墙之间，不能漂浮在房间中
- 家具和交互物不应直接用文字标注房间功能
- 房间功能通过家具和道具暗示
- 木牌应靠门设置，不应距离过远

### 3.4 农田与树木方向

- 树木是阻挡物，不能只是装饰点
- 树木当前太小，后续需要提升视觉体量
- 农田应成规模
- 田垄应有间隙，不应是一整块密铺
- 农田表现应更接近真实耕地纹理，而不是单纯一片底纹

## 4. 当前基准户型

当前已通过问答确认的基准样板为：

- 户型类别：普通贫农民居
- 居住人数：2 人
- 炊事方式：分体厨房
- 院落：要小院
- 院门方向：朝主路开
- 主屋结构：一明一暗
- 主屋逻辑：前屋起居，后屋睡眠，外接厨房

这个户型将作为后续：

- 纯蓝图验证
- 完整房屋效果预览
- 分层绘制规则
- 家具摆放合法性

的第一套标准样板。

## 5. 当前发现的主要问题

### 5.1 单层字符绘制造成结构被破坏

当前最关键问题不是符号是否好看，而是绘制规则错误。

前面多版预览中，以下内容被画在同一层：

- 墙
- 门
- 家具
- 玩家/NPC
- 名字
- 说明文字

结果导致：

- 墙体被人物名或说明字挤断
- 右侧墙比左侧更容易看起来不连通
- 门位不是从墙逻辑中生成，只是后加符号
- 家具可能落到墙上
- 人名可能显示到墙上

### 5.2 房屋布局先后顺序错误

之前的问题还有一层原因：

- 先摆家具和人物
- 再补墙和门

这个顺序会天然导致：

- 墙无法保持闭合
- 门位不合法
- 房间分区混乱
- 物件和人物可能放到墙上

正确顺序应改为：

1. 先设计房屋布局
2. 再定义被墙包裹的空间和功能分区
3. 再摆家具与交互物
4. 再放人物
5. 最后才放名字

### 5.3 墙体字符系统不统一

当前墙体仍存在：

- 单线和双线混用
- 拐角风格不统一
- 墙与门过渡逻辑不统一

用户要求统一为“双线墙”语义：

- 连续双线表示墙
- 门是墙中间的可开合构件
- 门打开时，横门变竖开门态，竖门变横开门态
- 打开后中间留出可通行空位

这意味着：

- 门不应再用单独的图标式符号替代墙逻辑
- 门位必须由墙体系统预留出来

### 5.4 路面与地表层逻辑不足

之前的主路/支路存在：

- 亮度过高
- 抢人物名字
- 视觉重叠
- 逻辑上不连贯

已确认后续方向：

- 主路不再用文字写“主路”
- 草地可以用绿色点状符号
- 土路可以用更低对比的土黄色点状纹理
- 地表表现要弱化，不能压过建筑和人物

## 6. 暂定代码设计

### 6.1 逻辑层与显示层分离

后续不能再直接“用字符代表全部逻辑”，而要拆成：

- 逻辑网格：定义每一格是什么、能不能走、有没有墙、有没有家具
- 显示网格：根据各层叠加结果，最终显示什么字符

### 6.2 暂定渲染分层

当前建议至少拆为以下 7 层：

1. 地表层
   - 草地、土路、主路、农田、水边
2. 结构层
   - 外墙、内墙、拐角、门位、院墙
3. 开合构件层
   - 门的开/关状态、木牌、可开合结构
4. 家具与交互物层
   - 床、桌、柜、灶台、箱子、铁砧、水井等
5. 角色实体层
   - 玩家、NPC、本体占位
6. 名字与状态层
   - 玩家名、NPC 两字名、选中状态、高亮
7. 交互提示层
   - 悬停、点击说明、调试信息

### 6.3 暂定碰撞判定

当前建议使用 tile-based 碰撞，而不是连续碰撞。

每格至少需要支持：

- `terrainType`
- `structureType`
- `doorState`
- `furnitureType`
- `isWalkable`
- `occupiedByActorId`

暂定规则：

- 墙格不可通行
- 关闭门不可通行
- 打开门可通行
- 树阻挡
- 大型家具默认阻挡
- 小型物件按类型决定是否阻挡
- 玩家和 NPC 不能站到墙上
- 名字不参与碰撞，但显示位置需要避让墙体

### 6.4 名字显示规则

人物名字不应再直接占用结构字符格。

正确方式应为：

- 人物实体有自己的占位格
- 名字作为上层 UI 标签显示
- 标签按锚点寻找安全显示位置
- 若当前位置会压墙，则切换显示锚点
