# AIRPGWeb 游戏内 Tile 地图编辑器设计

日期：2026-05-31
项目：`g:\UEGameDevelopment\Project\AIRPGWeb`
状态：设计已收敛，等待用户审阅

## 1. 设计目标

为 `AIRPGWeb` 增加一个游戏内置的开发者模式，并在其中提供首个可用模块：`地图绘制模块`。

本设计解决的不是单张房屋预览，而是更底层的问题：

- 用统一的 `tile` 数据和分层绘制规则替代手工字符拼接
- 让地图既可编辑、可保存、可回编，也能被游戏运行时直接读取
- 支持把多个局部地图拼接成一个更大的村庄世界
- 支持在编辑器里看到相邻地图的边界内容，避免接缝错位
- 支持 AI 基于用户已绘制的结构草稿，生成房屋布局草稿和指定层的风格化填充

首版优先服务于 `房屋/院落/村庄局部` 的布局与拓扑编辑，不覆盖地块经济、NPC 摆放编辑或完整世界流式加载。

## 2. 产品边界

### 2.1 首版包含

- 主界面新增 `开发者模式`
- 开发者模式内提供 `地图绘制` 模块
- 分层 tile 编辑
- 地图资产保存、打开、另存为
- 已绘制地图回编
- 多张子地图拼接为同一村庄世界
- 编辑时显示相邻地图的只读边界参考
- 将编辑结果设为运行时实际显示地图
- AI 生成房屋布局草稿
- AI 对指定层给出 `micro-pixel` 风格化美化结果

### 2.2 首版不包含

- 复杂地块数据系统
- NPC 行为编辑器
- 战斗/交互对象编辑器
- 完整自动地形生成
- 无限大世界 streaming
- 最终正式美术资源流水线

## 3. 关键设计原则

### 3.1 Tile 既承载显示，也承载数据

每个 tile 不只是一个显示符号或图片格，而是一个包含逻辑属性和视觉预设的基础单元。

但逻辑和显示不能硬绑在一个字段里，避免后续更换美术风格时破坏玩法数据。

### 3.2 逻辑网格与显示网格分离

- 逻辑网格决定区域、碰撞、结构、门位、连接关系
- 显示网格根据逻辑网格和当前风格 preset 渲染为字符 tile 或小像素块 tile

### 3.3 分层绘制，按层显示可绘制内容

用户切换到某一层时，右侧 palette 只显示该层允许绘制的 tile，减少画错层的概率。

### 3.4 先做多子地图拼接，再预留大世界扩展

首版采用“多张独立子地图拼接成村庄”的产品方案。

底层数据结构从一开始就保留 world/chunk 坐标和连接信息，为后续扩展成更大地图留接口。

### 3.5 AI 辅助不覆盖人工编辑

AI 首版的角色是：

- 生成草稿
- 生成风格化填充建议
- 生成指定层的可确认结果

AI 不能在未经确认的情况下直接覆盖用户正式编辑层。

### 3.6 结构拓扑采用手工精确模式

房屋拓扑编辑不采用“先随手画线、再自动猜角/T/十字”的主流程。

当前确认采用：

- 手工精确选择 tile family
- 手工精确放置直墙、角、T 字、十字、端头、门位 socket、开门、关门
- 编辑器负责做闭合、门位、层冲突和通行合法性校验

这样做的原因是：

- 你当前关注的是“户型图/施工图逻辑”
- 需要确保每一个墙角、门位和开门方向都可控
- 避免自动拓扑推断导致连通方式不符合预期

### 3.7 Tile 内容必须铺满整格

当前设计进一步确认：

- tile 内容必须铺满整个网格单元
- 相邻格子之间应尽量紧密贴合
- 网格线仅作为编辑参考叠加显示，不应依赖 `gap` 制造真实缝隙
- 美术覆盖层中的微纹理必须整格满铺，不能像悬浮在格子中央的小块

这条规则是为了解决当前测试中暴露出的两个问题：

- 墙线横向和纵向宽度不一致
- 微纹理无法铺满格子且格间距离过大

### 3.8 首版结构显示采用 DOM/CSS Tile Skin

首版结构层和美术覆盖层的显示，不再主要依赖字符本身的字宽与字体渲染结果。

当前推荐实现方式为：

- 每个 tile 仍然是标准网格单元
- 但 tile 内部使用 DOM/CSS 皮肤绘制横墙、竖墙、角、T 字、十字、门板、微纹理
- 字符仍可作为调试占位或 fallback，不作为最终对齐精度的唯一来源

这样可以更稳定地满足：

- 横墙双线等宽
- 竖墙双线等宽
- 门板占满整格
- 微纹理整格满铺
- 编辑网格紧贴显示

## 4. 开发者模式结构

开发者模式入口放在游戏主界面，进入后显示模块导航。

首版模块结构如下：

- 顶部工具栏
- 左侧层列表
- 中间网格画布
- 右侧当前层 tile 集
- 底部或侧边 AI 辅助面板

当前只启用 `地图绘制` 模块，但模块导航保留扩展位，方便后续增加其他开发工具。

## 5. 地图绘制模块结构

### 5.1 顶部工具栏

首版推荐包含以下工具：

- 画笔
- 橡皮
- 填充
- 矩形绘制
- 吸管
- 保存
- 另存为
- 打开地图
- 运行时预览
- AI 生成草稿

`撤销/重做` 结合开源编辑器调研结果，建议进入首版实施计划，不必作为第一批实现阻塞项，但不宜长期缺失。

进一步确认的交互规则：

- 单击：只绘制一个格子
- 长按并拖动：沿鼠标经过路径持续绘制
- 同一笔划重复经过同一格，不重复写入历史，避免撤销栈膨胀

对于结构层与开口层：

- 横向直墙拖刷时，应优先限制为横向连续写入
- 纵向直墙拖刷时，应优先限制为纵向连续写入
- 角、T 字、十字、门位 socket 这类精确拓扑 tile，默认仍以单击精确放置为主

### 5.2 左侧层列表

每层至少支持：

- 选中当前层
- 显隐切换
- 锁定切换
- 当前层高亮

用户选中某层后，只能编辑该层，并且右侧只显示该层允许使用的 tile 集。

### 5.3 中间网格画布

中间画布使用统一的游戏标准 tile 大小绘制。

当前设计不强制规定像素尺寸数值，但规定：

- 同一项目内 tile 尺寸必须统一
- 编辑器画布与游戏运行时渲染使用同一标准尺寸
- 逻辑网格和运行时渲染网格一一对应

当前已确认以下视觉约束：

- 画布格子之间尽量无真实缝隙
- 网格参考线采用叠加层显示，而不是使用格子间距撑开
- 结构和美术内容应先铺满整格，再叠加参考网格线

### 5.4 右侧 Tile Palette

右侧展示当前层可用的 tile family 和变体。

房屋结构首批 tile 集至少包含：

- 外墙直墙：横、竖
- 外墙角：左上、右上、左下、右下
- 内墙直墙：横、竖
- 内墙角：左上、右上、左下、右下
- 连接墙：T 上、T 右、T 下、T 左、十字
- 门位 socket：横、竖
- 门状态：开、关
- 院墙：横、竖、角、门位
- 房间区域标记 tile

本轮补充确认：

- 首版主模式为“手工精确模式”
- palette 中要明确区分：
  - `outer-wall-*`
  - `inner-wall-*`
  - `yard-wall-*`
  - `door-socket-*`
  - `door-leaf-*`
- 关门和开门都视为整格 tile，而不是嵌在线中间的半格装饰

### 5.5 AI 辅助面板

AI 面板支持两类输入：

- 结构草稿生成
- 指定层风格化美化

典型输入示例：

- “生成一个 2 人贫农民居布局图，带小院和分体厨房”
- “把当前结构层对应的室内地面改成木板 micro-pixel 风格”
- “给院地层补夯土地面纹理”

AI 输出默认进入：

- `AI 草稿层`
- 或 `AI 建议预览态`

用户确认后，才能合入正式层。

## 6. 分层顺序

首版地图绘制采用以下分层顺序：

1. 地表层
2. 区域层
3. 结构层
4. 开口层
5. 美术覆盖层

后续扩展预留：

6. 物件层
7. 交互标记层
8. 角色层
9. 标签层
10. 调试校验层

### 6.1 地表层

用于绘制：

- 草地
- 土路
- 院地
- 室内地面

### 6.2 区域层

用于标记功能区，不直接承担碰撞。

首版至少支持：

- `outside`
- `yard`
- `frontRoom`
- `sleepRoom`
- `storage`
- `kitchen`

### 6.3 结构层

用于绘制所有硬结构：

- 外墙
- 内墙
- 拐角
- T 字连接
- 十字连接

### 6.4 开口层

用于绘制与结构相关但可变的开口构件：

- 屋门
- 院门
- 门框
- 开门态
- 关门态

### 6.5 美术覆盖层

这是当前暂定的地图美术初版所在层。

该层不改变逻辑，只改变显示风格。

当前确认的视觉方向为：

- `Tile + 小像素块示意`
- 作为 `地图美术初版`
- 作为当前暂定视觉真相源

## 7. Tile 数据模型

首版建议使用如下逻辑结构：

```ts
type TileCell = {
  x: number
  y: number
  tileId: string
  visual: {
    family: 'wall' | 'door' | 'floor' | 'region' | 'overlay'
    variant: string
    style: 'blueprint' | 'microPixel'
  }
  logic: {
    region: 'outside' | 'yard' | 'frontRoom' | 'sleepRoom' | 'storage' | 'kitchen'
    terrain: 'grass' | 'dirtRoad' | 'yardFloor' | 'interiorFloor'
    structure: 'none' | 'outerWall' | 'innerWall' | 'corner' | 'junction' | 'doorSocket'
    orientation:
      | 'none'
      | 'h'
      | 'v'
      | 'tl'
      | 'tr'
      | 'bl'
      | 'br'
      | 't-up'
      | 't-right'
      | 't-down'
      | 't-left'
      | 'cross'
    doorState: 'none' | 'open' | 'closed'
    walkable: boolean
    blocksPlacement: boolean
  }
}
```

这意味着：

- 同一个 tile 既能显示，也能参与逻辑运算
- 更换视觉 preset 时，不需要重写碰撞和区域规则
- AI 可以直接读取和生成结构化布局

结合对 `Tiled` JSON 格式的调研，当前项目的地图资产格式建议继续向以下概念靠拢：

- `layers`
- `properties`
- `optional chunks`
- `objects`

这并不意味着首版需要兼容 TMX/TMJ，而是为了保证后续扩展到大地图、相邻地图显示和运行时桥接时，数据结构不会过早锁死。

## 8. 墙体与门的拓扑规则

墙体不能再只分“直墙”和“拐角墙”两类。

首版房屋拓扑最小集合必须包含：

- 直墙：横、竖
- 拐角：左上、右上、左下、右下
- T 字连接：上、右、下、左
- 十字连接
- 端头
- 门位 socket：横、竖

门的规则为：

- 门不是装饰 icon
- 门必须基于 `doorSocket` 生成
- 横墙中的门只能使用横门位
- 竖墙中的门只能使用竖门位
- 关门时阻挡通行
- 开门时门洞可通行

进一步确认：

- 门板占满整格
- 关门态与开门态都必须作为完整 tile 存在
- 不采用“门板压在线上”的表现方式，避免出现宽度不一致和格间断裂
- 门位逻辑仍然不能脱离结构层独立漂浮

后续即便视觉表现变化，门位逻辑也不允许脱离结构层独立漂浮。

## 9. 地图资产系统

### 9.1 MapAsset

每张独立可编辑地图保存为一个 `MapAsset`。

建议包含：

- 地图 id
- 名称
- 尺寸
- tile 层数据
- 默认出生点
- 地图入口点/出口点
- 缩略图或预览信息
- 是否为当前运行时地图

建议额外保留：

- `properties`
- `chunks` 或未来 chunk 扩展预留字段
- 编辑器版本号
- 风格 preset 信息

### 9.2 WorldAsset

`WorldAsset` 代表一个更大的世界容器，例如一个村庄。

它包含多张 `MapAsset` 的摆放关系和连接关系。

### 9.3 WorldPlacement

用于描述子地图在世界中的相对位置。

至少包含：

- `mapId`
- `worldX`
- `worldY`
- `width`
- `height`

### 9.4 MapLink

用于描述子地图之间的入口/出口连接关系。

例如：

- 某地图东侧出口通向另一张地图西侧入口
- 某院门切换到另一张子地图

## 10. 多地图拼接与相邻可见边界

你明确要求：多个已绘制地图在编辑时，相邻部分要可见，避免后续接不上。

这条被确定为首版设计要求。

首版方案如下：

- 中心画布编辑当前地图
- 对于相邻位置已有地图，编辑器以 `只读幽灵层` 显示它们的边界内容
- 幽灵层默认半透明，不可直接编辑
- 用户可切换“显示相邻地图参考”
- 用户可选择某张相邻地图后直接切换到该地图编辑

这个能力在开源调研中与 `Tiled` 的 world 视图和 `onlyShowAdjacentMaps` 思路高度一致，因此这里定义为硬需求，而不是可选增强项。

这样可解决：

- 路面接缝
- 院墙对接
- 地图边界断裂
- 房屋/广场/农田跨图时的错位

首版不支持跨多张图同时直接编辑，但要支持相邻边界可视参考。

## 11. 保存、打开、回编与运行时桥接

### 11.1 保存

首版必须支持：

- 保存当前地图
- 另存为新地图
- 保存当前世界布局

### 11.2 打开已有地图

用户必须可以从地图列表中打开已经绘制的地图继续修改。

### 11.3 设为运行时地图

编辑器不能只停留在工具态。

必须有明确的运行时桥接：

- 将当前地图或当前世界设为运行时使用对象
- 游戏运行时读取该对象进行实际显示

这样可以避免出现“编辑器里改了，但游戏里不是这张图”的混乱。

结合运行时桥接调研结果，建议在实现时明确抽出：

- `RuntimeMapLoader`
- `RuntimeWorldLoader`

用于把地图资产解析为游戏运行时可直接显示的结构，而不是把加载逻辑散落到现有世界显示组件中。

### 11.4 回编

已绘制好的地图必须可以再次进入编辑器，并按层继续修改。

AI 生成的草稿图同样要能回编，而不是一次性产物。

## 12. AI 辅助能力

AI 首版支持两类能力：

### 12.1 布局草稿生成

根据描述生成 `房屋/院落/村庄局部` 的结构草稿。

输出优先为结构化数据，不直接覆盖正式层。

### 12.2 指定层美化

当用户已画出基础结构后，AI 可以根据指定层生成视觉填充建议，例如：

- 地表层改成夯土地
- 室内地面改成木板 micro-pixel
- 院地换成更稀疏的颗粒土路

AI 在生成前，允许继续追问必要信息，例如：

- 这一层要用什么风格
- 目标区域是院子还是室内
- 希望偏木板、夯土还是石板

结合开源方案调研，当前不建议让 AI 直接输出“最终完整地图文件并覆盖当前编辑结果”。

首版更稳的方式仍然是：

- AI 输出结构化布局草稿
- 编辑器以草稿层或建议预览层显示
- 用户确认后合入正式层

## 13. 地图校验规则

保存前应执行基础校验。

首版建议包含：

- 外墙闭合检查
- 门位合法检查
- 地图边界检查
- 入口/出口合法检查
- 相邻地图连接检查
- 阻挡层与通行层冲突检查
- 运行时出生点合法检查

如果校验失败，应显示清晰提示并定位问题区域。

对手工精确模式，首版建议把以下规则视为高优先级校验项：

- `door-leaf-h-*` 只能放在 `door-socket-h`
- `door-leaf-v-*` 只能放在 `door-socket-v`
- 结构层不能在同一格同时出现互斥拓扑
- 开口层不能覆盖非法结构
- 锁定层不可被任何绘制工具修改

## 14. 运行时显示规则

运行时地图显示应直接读取编辑器保存后的地图资产或已发布世界资产。

首版要求：

- 游戏实际显示编辑结果
- 地图切换按 `MapLink` 或边界规则触发
- 运行时可继续使用当前选定的 micro-pixel 风格显示

这保证开发者模式不是孤立工具，而是游戏地图系统的真实前端入口。

## 15. 当前已确认的设计结论

截至当前，以下内容视为已确认：

- 使用分层 tile 编辑，而不是单层字符拼接
- `Tile + 小像素块示意` 作为当前暂定地图美术初版
- 开发者模式入口放在主界面
- 当前只先启用 `地图绘制` 模块
- 右侧 palette 随当前层切换
- 首版支持地图保存、打开、另存为、回编
- 首版支持多张子地图拼接成村庄
- 底层为后续大世界/chunk 扩展预留接口
- 编辑时允许看到相邻地图的只读边界参考
- 游戏运行时使用编辑器生成的地图资产
- AI 用于草稿生成和指定层美化，不直接无确认覆盖正式层

## 16. 实施建议

建议按以下顺序进入实现规划：

1. 开发者模式壳层与地图绘制模块 UI
2. tile 数据模型与分层编辑基础
3. 房屋拓扑 tile 集与 palette
4. 地图保存/打开/回编
5. 世界拼接与相邻地图幽灵层
6. 运行时地图加载桥接
7. AI 草稿生成与层级美化

这个顺序既能尽快看到可用编辑器，又不会过早引入过重的 AI 或世界系统复杂度。

## 18. 交互增强分批建议

### 18.1 第一批优先实现

- 手工精确 tile family palette 扩展
- DOM/CSS tile skin，解决整格铺满与线宽统一
- 单击绘制一个格子
- 长按拖刷连续路径
- 图层锁定
- 地图命名
- 另存为
- 世界布局拖拽与网格吸附

### 18.2 第二批实现

- 矩形绘制
- 填充
- 吸管
- 更复杂的拓扑辅助校验 UI
- 批量选区操作

### 18.3 当前分批原因

第一批的目标不是“做最多工具”，而是优先把你当前最核心的户型图绘制问题解决：

- 墙门严丝合缝
- 微纹理铺满整格
- 手工精确控制房屋拓扑
- 拖刷能快速铺一排墙或一片地面

## 17. 开源调研吸收结论

本设计已吸收以下开源项目的成熟思路：

- `mapeditor/tiled`
  - 吸收：`world` 组织方式、相邻地图显示、图层和属性组织、Terrain/Wang 的规则化思想
  - 不照搬：完整 TMX/TMJ 兼容和全部对象编辑能力
- `blurymind/tilemap-editor`
  - 吸收：轻量 Web 编辑器的工具栏、图层、导入导出、模块嵌入思路
  - 不照搬：其偏通用画布的弱拓扑约束
- `wareya/autotyler`
  - 吸收：autotile 的规则族思路，说明必须先定义墙/门/连接 family
  - 不照搬：当前阶段不过早引入完整自动规则生成
- `excaliburjs/excalibur-tiled`
  - 吸收：运行时资源读取与场景桥接思路
  - 不照搬：引擎绑定实现
