# AIRPGWeb 存档修复与随机名字设计

日期：2026-05-31
项目：`g:\UEGameDevelopment\Project\AIRPGWeb`
状态：已确认，待用户审阅

## 1. 目标

修复 `AIRPGWeb` 当前主流程里两个核心问题，并补齐角色创建阶段的随机名字能力：

1. 本地已有存档时，主菜单 `继续游戏` 仍可能是灰色不可点
2. 游戏内加载存档后，玩家位置没有稳定恢复或缺少明确可见证据
3. 角色创建里，玩家姓名和青梅竹马姓名输入框后需要增加符合世界观的随机名字按钮

本次设计目标不是重写整套前端架构，而是在保留当前页面结构的前提下，统一存档状态判断、收敛位置恢复链路、补强验证，并新增数据驱动的随机名字方案。

## 2. 范围

本次范围内：

- 启动时同步本地存档状态，正确决定 `继续游戏` 是否可用
- 统一主菜单、游戏内、返回主菜单等路径对存档存在性的刷新逻辑
- 修复自动存档与手动存档的写入语义，避免错误覆盖出生坐标
- 明确读档后的运行时恢复链路，保证坐标、时间、季节、天气恢复一致
- 在游戏界面提供可见的坐标/读档反馈，避免“实际上恢复了但看起来没变化”
- 为角色创建增加主角与青梅竹马的随机名字按钮
- 补充针对“刷新后继续游戏”和“加载后位置恢复”的自动化测试

本次范围外：

- 扩展更多存档字段，如任务状态、NPC 记忆、探索区域、村庄稳定度
- 调整主菜单 `继续游戏` 语义为“直接读取最近存档”
- 重做整个游戏界面布局或文本地图表现形式
- 引入全局状态库对整个应用状态做彻底重构

## 3. 当前问题与根因

### 3.1 继续游戏灰色

当前 `App.tsx` 中：

- `hasSave` 初始值固定为 `false`
- 应用启动时没有读取 IndexedDB 初始化 `hasSave`
- 只有部分退出路径才会重新调用 `getAllSlots()`

这意味着即使本地数据库里已经有 `auto-save` 或 `slot-1/2/3`，刷新页面或重新打开应用后，主菜单仍会把 `继续游戏` 渲染为禁用状态。

### 3.2 自动存档被错误覆盖

当前用于“从设置返回主菜单并选择保存”的逻辑，会调用一个只适合“新建初始角色快照”的 helper，导致：

- 世界时间被重置为固定值
- 天气和季节被重置为固定值
- 玩家坐标被写回出生点

因此部分自动存档虽然保存成功，但内容可能已经退化成初始状态，进一步放大“加载后位置没变化”或“位置不对”的问题。

### 3.3 加载后位置恢复缺少稳定证据

当前坐标状态分布在三层：

- `App` 持有 `loadedState.playerX/playerY`
- `GameShell` 持有运行时 `playerX/playerY`
- `TextMapView` 再持有内部 `playerX/playerY`

读档流程虽然会通过 `gameKey` 强制重挂载游戏壳，但由于文本地图视口始终围绕玩家居中，读档前后画面可能看起来相似，用户很难确认坐标是否真正恢复。

当前测试也没有断言“保存时坐标”和“读档后坐标”完全一致，因此存在“流程通过但问题仍然存在”的假阳性。

### 3.4 随机名字能力缺失

当前角色创建只有自由输入姓名，没有任何名字池或名字生成逻辑。若直接把随机数组写在组件里，会把世界观命名规则与 UI 绑死，后续不好扩展。

## 4. 设计原则

- 保持纯文本表现路线，不把世界观命名和存档规则硬编码到 UI 细节中
- 数据层先行：名字池与存档规则放在数据/应用层，组件只负责调用
- 单一职责：主菜单是否可继续，由统一的存档状态刷新函数决定
- 明确真源：读档恢复时，以单一快照对象作为输入，不在多处各自猜测默认值
- 证据优先：修复后必须让用户看得到坐标与读档结果，测试也必须能断言结果

## 5. 方案概览

采用“保留现有页面状态机，补一个统一存档服务层用法约束”的方案。

### 5.1 统一存档存在性判断

在 `App.tsx` 中新增统一函数：

- `refreshSaveAvailability()`

职责：

- 读取 `save-repository.getAllSlots()`
- 过滤出存在有效 `savedAt` 的槽位
- 统一设置 `hasSave`
- 返回当前槽位列表，供调用方复用

调用时机：

- 应用首次挂载时
- 新游戏自动存档完成后
- 手动保存完成后
- 保存并退出后
- 不保存退出后
- 从设置返回主菜单后
- 未来删除存档后

规则：

- 只要 `auto-save` 或任一手动槽位存在有效存档，`继续游戏` 就可用

### 5.2 区分“初始自动存档”与“运行时自动存档”

当前 `autoSlot()` 只适合“角色创建完成后第一次进入世界”。

调整为两个 helper：

- `buildInitialAutoSave(player)`：仅用于角色创建后第一次自动存档
- `buildRuntimeSnapshot(slotId, player, x, y, day, minute, season, weather)`：用于运行中手动保存、退出自动保存、设置里保存并返回

规则：

- 任何运行中保存都必须使用当前实时坐标与世界状态
- 不允许再复用“初始存档 helper”去覆盖运行时自动存档

### 5.3 统一读档快照输入

保留 `App` 为唯一读档入口，但明确约束：

- `loadSlot(slotId)` 只负责记录待加载槽位并进入加载界面
- `handleLoadComplete()` 只负责从仓储读取指定槽位，并把完整快照写入 `loadedState`
- `GameShell` 只接收一份明确的初始游戏快照，不再自行猜默认值

建议将当前读档输入组织为单一对象：

- `initialGameState.player`
- `initialGameState.playerX`
- `initialGameState.playerY`
- `initialGameState.worldDay`
- `initialGameState.worldMinute`
- `initialGameState.season`
- `initialGameState.weather`

这样后续如果继续扩展任务状态、地图探索等，也能沿同一快照结构前进。

### 5.4 增加读档后的可见证据

为了避免“位置恢复了但用户看不出来”，游戏界面增加两类可见反馈：

1. 常驻坐标显示
2. 读档成功提示

第一版建议：

- 在 `WorldInfo` 或 `GameShell` 中显示当前坐标，例如：`坐标：(24, 17)`
- 读档成功后在日志中写入：`已加载 存档槽 1，位置 (24, 17)，第 1 天 08:00`
- 存档面板列表可追加一行简要详情：玩家名、坐标、时间

这样既能帮助用户确认，也能为 Playwright 提供稳定断言点。

### 5.5 随机名字的数据驱动方案

新增名字数据文件，例如：

- `src/data/names/given-names.ts`
- `src/domain/names/random-name.ts`

第一版名字规则：

- 按性别分为 `male` / `female` 两组
- 风格采用“中世纪村镇 fantasy 中文译名风格”
- 名字以 2-3 字为主，优先朴素、日常、乡镇感，不走仙侠或现代网名风格

组件交互：

- 玩家姓名输入框右侧增加 `随机名字`
- 青梅竹马姓名输入框右侧增加 `随机名字`
- 若已选择性别，则优先从对应性别池抽取
- 若尚未选择性别，则从中性合并池抽取，或默认先用全部池抽取

第一版推荐名字池结构：

- `maleNames`
- `femaleNames`
- `neutralFallbackNames`（可选）

第一版不强制去重，但随机时应尽量避开另一个角色当前已经使用的名字，减少主角与青梅竹马重名概率。

## 6. 数据流设计

### 6.1 应用启动

1. `App` 挂载
2. 调用 `refreshSaveAvailability()`
3. 读取全部本地槽位
4. 若存在有效存档，则设置 `hasSave = true`
5. 主菜单据此决定 `继续游戏` 是否可点击

### 6.2 角色创建后自动存档

1. 用户完成角色创建
2. 进入加载界面
3. 构建玩家初始数据
4. 使用 `buildInitialAutoSave(player)` 写入 `auto-save`
5. 存档完成后刷新 `hasSave`
6. 进入游戏

### 6.3 游戏内手动保存

1. 用户打开存档面板
2. 选择手动槽位
3. `GameShell` 把当前实时 `x/y/day/minute/season/weather` 传给上层
4. `App` 使用 `buildRuntimeSnapshot(slotId, ...)` 写入数据库
5. 刷新槽位信息与 `hasSave`

### 6.4 游戏内加载

1. 用户在游戏内或主菜单选择某个槽位
2. 应用进入加载界面
3. 加载完成后从仓储读取槽位快照
4. 生成新的 `initialGameState`
5. 使用 `gameKey` 强制重挂载 `GameShell`
6. 游戏内显示坐标和读档成功日志

### 6.5 返回主菜单并保存

1. 用户在游戏内点击返回主菜单
2. 弹出确认框
3. 选择保存并返回时，必须使用当前运行时快照覆盖 `auto-save`
4. 刷新 `hasSave`
5. 回到主菜单

## 8. 具体实现约束

### 8.1 存档存在性

- 不能再依赖局部流程临时设置 `hasSave`
- 不能以 `slots.length > 0` 代替“是否有有效存档”的判断，必须确认槽位确实存在存档记录

### 8.2 快照构建

- 运行时快照必须显式写入：
  - `slotId`
  - `savedAt`
  - `playerName`
  - `player`
  - `worldDay`
  - `worldMinute`
  - `season`
  - `weather`
  - `playerX`
  - `playerY`
- 不允许用固定出生坐标覆盖运行时位置

### 8.3 读档验证

- 读档成功后必须能从 UI 上直接看出当前坐标
- 不能只依赖“人物似乎在地图中央”来判断恢复成功

### 8.4 随机名字

- 名字数据与 UI 分离
- 不在组件里硬编码大量名字文本
- 第一版名字池保持小而精，后续可扩充，不做复杂姓氏组合器

## 10. 风险点

- 当前 IndexedDB 中可能已经存在被错误 helper 覆盖的旧自动存档，修复后需要考虑是否接受历史脏数据继续存在
- 如果仍然保留三层位置状态而不约束职责，未来继续扩展地图切换或睡觉存档时还会反复出问题
- 若存档面板增加过多字段，文本 UI 可能显得拥挤，第一版应只展示最关键的信息
- 名字池如果过小，频繁点击随机会出现重复感，需要后续按地区或身份扩容

## 11. 缺失信息与默认假设

当前先采用以下默认假设推进：

- `继续游戏` 的语义保持为“进入存档选择界面”
- 存档列表按现有固定槽位展示，不改成最近存档列表
- 名字风格采用“中世纪村镇 fantasy 中文译名风格”
- 随机命名允许重复，但尽量避开主角与青梅竹马当前重名

若后续要改成：

- `继续游戏` 直接读最近存档
- 名字改为更明显的西幻音译风格
- 引入姓氏 + 名字拼接

则应作为下一轮独立设计调整，不和本次修复混在一起。

## 12. 实现顺序建议

1. 先修 `App.tsx` 的存档存在性初始化和快照 helper
2. 再补游戏内坐标显示与读档成功证据
3. 然后补 Playwright 坐标断言测试
4. 最后增加随机名字数据文件与角色创建按钮

这样可以优先解决主流程 bug，再补输入体验，不会把两个问题耦在一起难以定位。
