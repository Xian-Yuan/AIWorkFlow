# 框架自主成长机制（Autonomous Growth Framework）

## 目标

框架**不需要等你说"自我完善"**。它在日常开发中持续自检、发现机会、向你汇报、经你批准后自我升级。

---

## 四大引擎

```
┌─────────────────────────────────────────────────────────┐
│ 引擎 1: 持续自检 (Continuous Audit)                      │
│   每次会话自动运行 3 脚本，检测健康状态                     │
│   触发: 会话开始 / 创建 Docs/ 文件 / 每日                                             │
├─────────────────────────────────────────────────────────┤
│ 引擎 2: 被动学习 (Passive Learning)                      │
│   开发中遇到新错误/新模式/新知识 → 自动沉淀                     │
│   触发: 每次编译修复 / 每次新类创建 / 每次概念讨论      │
├─────────────────────────────────────────────────────────┤
│ 引擎 3: 主动发现 (Active Discovery)                      │
│   连续 2 次同类问题 or 用户困惑 → 联网搜索 → 汇报          │
│   触发: 会话中检测到重复困难 / 用户说"不知道怎么做"     │
├─────────────────────────────────────────────────────────┤
│ 引擎 4: 定期升级 (Scheduled Upgrade)                     │
│   用户说"自主成长"时，执行全量自检 + 联网搜索 + 汇报       │
│   触发: "自主成长" / "应该学习什么" / "有什么可以改进"   │
└─────────────────────────────────────────────────────────┘
```

---

## 引擎 1: 持续自检 (Continuous Audit)

### 原则：只在需要时检查，不做无意义扫描

| 场景 | 执行 | 开销 |
|------|------|:---:|
| **创建/重命名 `Docs/NN-*.md`** | `check-numbering -Fix` | ~0.1s |
| **创建新文件在非标准位置** | `check-placement` | ~1s |
| **用户说"自主成长"** | 3 脚本全跑 | ~3s |
| **会话开始** | ❌ 不跑 — 无变化时不检查 | - |
| **普通文件修改** | ❌ 不跑 — 不涉及编号 | - |

### 轻量替代：自检摘要

会话开始时，只做一次**零开销检查**：
```
统计 Docs/ 文件数 → 与上次记录的差值 → 如有新增，提醒运行 check-numbering
```
这条信息本身不需要读取任何文件，只需对比一个数字。

## 引擎 2: 被动学习 (Passive Learning)

### 自动沉淀规则

| 触发条件 | 自动行为 | 需要确认？ |
|---------|---------|:---:|
| 修复了一个 ErrorKB 中不存在的编译错误 | 用 E000-TEMPLATE.md 创建新条目 | ❌ 自动 |
| 修复了一个 ErrorKB 中不存在的运行时错误 | 同上 | ❌ 自动 |
| 实现了一个新的可复用代码模式 | 追加到最相关 Skill 末尾 | ⚠️ 标记，下次"自主成长"时确认 |
| 发现 Docs/ 中过时信息 | 在文档末尾追加 `⚠️ Deprecated: ...` | ❌ 自动 |
| 用户纠正了一次错误路由 | 更新 Router Agent 路由表 | ❌ 自动 |
| 使用了一个新插件/新模块 | 记录到框架健康日志 | ❌ 自动 |
| 发现 Skill 中缺少某个领域知识 | 标记为缺失，下次"自主成长"时搜索 | ⚠️ 标记 |

### 不自动创建的情况

- 模式太小（< 10 行代码）
- 太特殊（仅当前项目用一次）
- 用户表示"这只是临时方案"

---

## 引擎 3: 主动发现 (Active Discovery)

### 触发条件

| 场景 | 行为 |
|------|------|
| 同一类问题在会话中出现第 2 次 | "这个问题之前也出现了，我搜索一下更好的方案" → 联网搜索 |
| 你表达了困惑（"不知道怎么做" / "有没有更好的办法"） | 自动触发搜索 |
| 遇到 Docs/ APIRef/ 无法覆盖的 API | 自动标记缺失，下次"自主成长"时补 |
| 使用了一个新概念（之前未在 Skill/Docs 中出现） | 记录，后续评估是否需要新建 Skill |

### 搜索策略

1. 优先搜索 Epic 官方文档
2. 再搜索 Unreal Slackers / Reddit / 社区
3. 最后搜索通用方案（Claude Code / OpenCode 最佳实践）

---

## 引擎 4: 定期升级 (Scheduled Upgrade)

### 触发词

你说以下任一即触发全量升级：
- "自主成长"
- "应该学习什么"
- "有什么可以改进"
- "检查框架"
- "提升自己"

### 流程（必须全部执行，不可跳过）

```
1. 运行 3 个脚本（check-numbering -Fix + check-placement + check-dead-code）
2. 扫描被动学习标记（新模式 / 缺失知识）
3. ★ 联网搜索可提升框架的方案（必须执行，搜索方向见下表）
4. 汇总汇报（含优先级 + 利弊分析）
5. 等你批准后执行
```

### 联网搜索方向（每次"自主成长"至少搜索 2 个方向）

| 方向 | 搜索目标 | 用途 |
|------|---------|------|
| Claude Code / OpenCode 最新功能 | Agent Team、Hook、Skill 增强、插件 | 提升 Agent 协作和自动化 |
| AI 编码最佳实践 | 上下文管理、防幻觉、提示工程 | 让 AI 更准确 |
| UE5 工具/插件/MCP | 编辑器自动化、蓝图辅助、资产管线 | 提升 UE 开发效率 |
| 知识管理方案 | 文档索引自动化、知识图谱、向量检索 | 提升 Docs/ 查询效率 |
