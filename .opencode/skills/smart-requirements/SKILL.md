# Smart Requirements Clarification

## 定位

Plan 阶段启动时激活，确保 Agent 在动手之前充分理解用户真正想要什么。核心原则：**Ask twice, implement once.**

## 何时调用

- **Plan 阶段 Step 1c**: `ue-project-router` 在做路由决策前必须运行
- **用户需求模糊不清时**: 需求中缺少明确的技术边界、目标系统、验收标准
- **用户只说结果不说路径时**: "做个角色系统" → 需要追问角色包含什么

## 规则 1：5W 需求深挖

遇到模糊需求时，至少追问 3 层：

```
Layer 1: What（做什么）→ 输出什么结果？
Layer 2: Why（为什么）→ 解决什么真实问题？
Layer 3: Who/How（谁/怎么）→ 谁用、怎么用、什么约束？
```

禁止在 Layer 1 就停下来开始写代码。

### 示例
输入: "给角色加个技能系统"

```
L1 What: 角色通过输入触发技能，技能有冷却和消耗
L2 Why:  战斗需要多样化攻击方式，不只是普攻
L3 How:  通过 Lyra/GAS InputConfig → AbilitySet → GameplayAbility 链条
         单机，不需要网络同步
         技能需要蓝耗、CD、伤害类型的 GE
```

## 规则 2：隐性需求推导

用户没说出口但必须实现的：

| 需求类别 | 隐性前提 | 检查项 |
|---------|---------|--------|
| 功能 | "加个技能" | 技能图标、音效、粒子特效、UI 冷却条 |
| 数据 | "角色有血量" | 血量用 GE Modifier 还是 AttributeSet 默认值 |
| 交互 | "按E键交互" | 需要 SmartObject？需要交互提示 UI？ |
| 异常 | "角色攻击敌人" | 敌人不存在怎么办？敌人已死？距离不够？ |
| 扩展 | "做一个BUFF系统" | 未来是否需要堆叠？驱散？免疫？ |

## 规则 3：确认型提问

以下情况必须先确认再动手：

1. **技术方案有 2+ 选择**: 展示选项 + 推荐 + 理由，等用户选择
2. **实现成本差异大**: "方案A（标准Lyra路径）vs 方案B（快速Blueprint路径），A需要3小时但可扩展，B只需30分钟但后期重构成本高"
3. **影响已有系统**: "这个改动会影响到现有的 X 系统，是否确认？"

## 规则 4：需求完整性检查清单

Plan 阶段输出前，逐项检查：

- [ ] 功能目标已明确
- [ ] 技术路线已选定
- [ ] 依赖项已识别
- [ ] 隐性需求已列出
- [ ] 异常路径已覆盖
- [ ] 验收标准已定义（spec.md）
- [ ] 用户已确认（不可跳过）

## 禁止事项

- 不在需求模糊时直接开始实现
- 不为用户做假设（"用户应该是要 X"）
- 不问用户可以通过上下文推断的问题
- 不跳过用户确认步骤

## 与现有 Skill 集成

- 扩展 `implicit-requirements`：加入交互/异常/扩展维度
- 增强 `ue-project-router` Step 1c：每个模糊信号都触发 Rule 1
