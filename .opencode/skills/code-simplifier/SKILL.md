# Code Simplifier — 代码简化

## 原则

**目标不是减少行数，而是让代码更容易被理解、修改和调试。**

每次简化必须通过一个测试："新成员理解这段代码比原来更快吗？"

## 何时使用

- 功能已实现且测试通过，但实现感觉比需要的更重
- Code Review 时发现可读性或复杂度问题
- 遇到深层嵌套逻辑、过长函数、不清晰的命名
- 合并变更后引入重复或不一致

## 五个原则

### 1. 保持行为完全不变
不改变代码做什么——只改变它如何表达。所有输入、输出、副作用、错误行为、边界情况必须完全相同。

### 2. 遵循项目约定
简化意味着让代码更符合代码库风格，而非强加外部偏好。先读项目编码规范，研究相邻代码风格。

### 3. 清晰优先而非聪明
```javascript
// ❌ 不清晰：密集三元链
const label = isNew ? 'New' : isUpdated ? 'Updated' : isArchived ? 'Archived' : 'Active';

// ✅ 清晰：可读映射
function getStatusLabel(item) {
  if (item.isNew) return 'New';
  if (item.isUpdated) return 'Updated';
  if (item.isArchived) return 'Archived';
  return 'Active';
}
```

### 4. 保持平衡
避免过度简化——不能降低清晰度、不能合并过多关注点到单一函数、不能删除有帮助的抽象。

### 5. 限定范围
默认只简化当前会话最近修改的代码。

## 常见简化模式

| 问题 | 简化 |
|------|------|
| 深层嵌套 if | 提前返回、卫语句 |
| 过长函数（>50行） | 提取子函数，单一职责 |
| 重复逻辑 | 提取共享函数 |
| 魔法数字/字符串 | 命名常量 |
| 多层循环嵌套 | 提取内层为函数 |
| 不清晰命名 | 重命名为描述性名称 |
| 无意义注释 | 删除注释，让代码自解释 |
| 不必要抽象 | 合并过于分散的类/函数 |
