# 横向专题：异步执行模式

## 概述

UE 提供了从低级到高级、从独立线程到任务编排的完整异步执行体系。第 17~18 章依次介绍了五种异步模式，按抽象层级从低到高排列：

| 模式 | 抽象层级 | 线程模型 | 适用场景 |
|------|---------|---------|---------|
| FRunnable 独立线程 | 最低（OS 线程裸封装） | 独立线程（非线程池） | 长驻后台线程、持续运行的 Worker |
| AsyncTask / Async | 中低 | TaskGraph 线程池 | 一次性轻量异步任务、需要返回值的异步调用 |
| FGraphEvent | 中 | TaskGraph 线程池 | 带依赖关系的任务 DAG |
| ParallelFor | 中 | TaskGraph 线程池 / 单线程 | 数据并行、纯计算密集型 |
| ControlFlows | 高（异步编排框架） | 内部使用 TaskGraph 线程池 | 多步骤异步流程编排、进度管理 |

---

## 一、FRunnable 独立线程（第 17 章）

### 1.1 接口定义

FRunnable 是 UE 中最底层的线程抽象，直接封装 OS 线程。核心声明见：

```cpp
// 参考：XGSimpleRunnable.h
class FXGSimpleRunnable : public FRunnable
{
    virtual bool Init() override;
    virtual uint32 Run() override;
    virtual void Exit() override;
    virtual void Stop() override;

    FString XGSimpleRunnableThreadName;
    bool bRunning = true;
    FCriticalSection CriticalSection;
};
```

### 1.2 生命周期

| 方法 | 调用线程 | 时机 | 行为 |
|------|---------|------|------|
| `Init()` | 新线程 | 线程启动后、Run 之前 | 初始化资源，返回 false 终止线程 |
| `Run()` | 新线程 | Init 成功后 | 主循环，返回 0 后线程终止 |
| `Stop()` | **外部调用方线程** | 外部请求停止 | 设置退出标志，不阻塞等待 |
| `Exit()` | 新线程 | Run 返回后 | 清理资源 |

启动方式：

```cpp
// 参考：XGThreadSubsystem.cpp
FRunnableThread* RunnableThread = FRunnableThread::Create(
    XGSimpleRunnable.Get(),
    *XGSimpleRunnable->GetThreadName()
);
```

关键特性：
- `Stop()` 在**外部线程**执行，不阻塞。取中逻辑是设置标志后，Run 内循环自行检测退出
- `Run()` 内的循环通常配合 `FPlatformProcess::Sleep()` 控制轮询频率，避免空占 CPU
- 非线程池管理，需要调用方自行持有 `FRunnableThread*` 指针

### 1.3 线程安全实现对比

| 模式 | 声明 | 原理 | 使用场景 |
|------|------|------|---------|
| `FCriticalSection + bool` | `FCriticalSection CriticalSection; bool bRunning;` | 手动加锁保护读写 | 需要保护多个变量的复合操作 |
| `FThreadSafeBool` | `FThreadSafeBool bRunning = true;` | 原子操作（`std::atomic<bool>`） | 仅需保护单个 bool 标志位 |

```cpp
// FCriticalSection 模式（XGSimpleRunnable.cpp）
void FXGSimpleRunnable::Stop()
{
    FScopeLock Lock(&CriticalSection);
    bRunning = false;
}
```

```cpp
// FThreadSafeBool 模式（XGSimpleRunnableBool.h/.cpp）
FThreadSafeBool bRunning = true;

// Run 中直接用 bRunning 作为 while 条件
while (bRunning)
{
    FPlatformProcess::Sleep(0.04);
    if (!bRunning) break;  // 双重检查
    // 工作
}

// Stop 中直接赋值
void FXGSimpleRunnableBool::Stop()
{
    bRunning = false;
}
```

**选型建议**：如果 FRunnable 内只有一个 bool 标志需要共享，用 `FThreadSafeBool` 更轻量；如果有多个变量需要一致地读写，用 `FCriticalSection`。

## 二、AsyncTask / Async 轻量异步（第 17 章）

### 2.1 AsyncTask（无返回值）

```cpp
// 参考：XGThreadSubsystem.cpp
AsyncTask(ENamedThreads::GameThread, []() {
    PrintWarning(...);
});

AsyncTask(ENamedThreads::AnyThread, [this, CopyNum]() {
    // 在线程池中执行
    // 工作...
    AsyncTask(ENamedThreads::GameThread, [this, Max]() {
        SetMyNum(Max);  // 跳回主线程访问 UObject
    });
});
```

### 2.2 Async + TFuture（有返回值）

```cpp
// 参考：XGThreadSubsystem.cpp
TArray<TFuture<int32>> Results;

for (int32 i = 0; i < 4; i++)
{
    Results.Add(Async(EAsyncExecution::Thread, [i]() -> int32 {
        int32 SleepTime = (i + 1) * 3;
        FPlatformProcess::Sleep(SleepTime);
        return SleepTime;
    }));
}

// 阻塞等待所有结果
for (auto& Future : Results)
{
    int32 Result = Future.Get();
}
```

### 2.3 线程跳跃模式

最常用的异步模式是**从任意线程跳回 GameThread**：

```cpp
// 任何线程 → 工作 → GameThread
AsyncTask(ENamedThreads::AnyThread, [this]() {
    // 耗时计算（线程池）
    
    AsyncTask(ENamedThreads::GameThread, [this]() {
        // 访问 UObject / GameThread 资源
        SetMyNum(42);
    });
});
```

```cpp
// SetMyNum 自身实现线程安全（参考 XGThreadSubsystem.cpp）
void UXGThreadSubsystem::SetMyNum(int32 InInt)
{
    if (!IsInGameThread())
    {
        AsyncTask(ENamedThreads::GameThread, [this, InInt]() {
            SetMyNum(InInt);
        });
        return;
    }
    MyIntNum = InInt;
}
```

## 三、FGraphEvent 任务依赖 DAG（第 17 章）

> 虽然叫 GraphEvent，但本质是 **TaskGraph 的依赖任务系统**，不应与 UObject 的 FEvent 或跨平台 FEvent 混用。

### 3.1 单任务

```cpp
// 参考：XGThreadSubsystem.cpp
FGraphEventRef Event = FFunctionGraphTask::CreateAndDispatchWhenReady(
    []() {
        FPlatformProcess::Sleep(1.0);
    }
);
Event->Wait();  // 阻塞等待
```

### 3.2 批量任务

```cpp
// 参考：XGThreadSubsystem.cpp
FGraphEventArray Events;
for (int32 i = 0; i < 5; i++)
{
    Events.Add(FFunctionGraphTask::CreateAndDispatchWhenReady(
        [i]() {
            FPlatformProcess::Sleep(0.2 * (i + 1));
        }
    ));
}

FTaskGraphInterface::Get().WaitUntilTasksComplete(Events);
```

### 3.3 依赖链（DAG）

```cpp
// 参考：XGThreadSubsystem.cpp

// 任务 A（无依赖）
FGraphEventRef A = FFunctionGraphTask::CreateAndDispatchWhenReady(
    []() { /* A */ }
);

// 任务 B 依赖 A
FGraphEventRef B = FFunctionGraphTask::CreateAndDispatchWhenReady(
    []() { /* B */ },
    TStatId(), nullptr, ENamedThreads::GameThread,
    { A }  // Prerequisites
);

// 任务 C 依赖 B
FGraphEventRef C = FFunctionGraphTask::CreateAndDispatchWhenReady(
    []() { /* C */ },
    TStatId(), nullptr, ENamedThreads::GameThread,
    { B }  // Prerequisites
);

// A → B → C（串行链）

// ===== 也可以做并行分支 =====

// A → A1（并行支线，依赖 A）
FGraphEventRef A1 = FFunctionGraphTask::CreateAndDispatchWhenReady(
    []() { /* A1 */ },
    TStatId(), nullptr, ENamedThreads::AnyThread,
    { A }  // 依赖 A
);

// D 依赖 A1 + C
FGraphEventRef D = FFunctionGraphTask::CreateAndDispatchWhenReady(
    []() { /* D */ },
    TStatId(), nullptr, ENamedThreads::GameThread,
    { A1, C }  // 多个先决条件
);
```

**DAG 关系图**：
```
     A
    / \
   B   A1
   |    |
   C    |
    \  /
     D
```

## 四、ParallelFor 并行循环（第 17 章）

### 4.1 基本使用

```cpp
// 参考：XGThreadSubsystem.cpp
ParallelFor(10, [](int32 Index) {
    FPlatformProcess::Sleep(0.2);
});
```

### 4.2 标志组合

```cpp
// 常用的组合
ParallelFor(10, [](int32 Index) {
    // 工作...
}, EParallelForFlags::ForceSingleThread     // 单线程（调试用）
   | EParallelForFlags::BackgroundPriority  // 后台优先级
   | EParallelForFlags::Unbalanced);        // 非均衡分配（适用于各迭代耗时差异大）
```

| 标志 | 作用 |
|------|------|
| `ForceSingleThread` | 强制单线程执行，用于调试线程安全问题 |
| `BackgroundPriority` | 后台优先级，不干扰前台渲染 |
| `Unbalanced` | 非均衡分配，每次迭代取下一个工作项（而非预先均分范围），适用于各次迭代耗时差异大的场景 |

### 4.3 共享变量保护

```cpp
// 参考：XGThreadSubsystem.cpp
FCriticalSection NumCriticalSection;
int32 Num = 0;

ParallelFor(10, [&Num, &NumCriticalSection](int32 Index) {
    FPlatformProcess::Sleep(0.2);
    {
        FScopeLock Lock(&NumCriticalSection);
        Num += Index;
    }
});
```

## 五、ControlFlows 异步编排（第 18 章）

> ControlFlows 是 UE5 的实验性插件（`ControlFlows`），提供一种声明式的异步任务编排方式。通过 `FControlFlowStatics` 创建流程，`QueueStep` 注册步骤，内部使用 TaskGraph 线程池执行。

### 5.1 基本流程

```cpp
// 参考：XGControlFlowsSubsystem.cpp

// 1. 创建流程
FControlFlow& Flow = FControlFlowStatics::Create(this, TEXT("XGControlFlowInitLevel"));

// 2. 注册步骤（顺序执行）
Flow.QueueStep(TEXT("InitLocalAsset"),       this, &UXGControlFlowsSubsystem::InitLocalAsset,  0.1);
Flow.QueueStep(TEXT("InitNetInfo"),          this, &UXGControlFlowsSubsystem::InitNetInfo,     0.2);
Flow.QueueStep(TEXT("InitUserInfo"),         this, &UXGControlFlowsSubsystem::InitUserInfo,    0.5);
Flow.QueueStep(TEXT("NotifyMainUI"),         this, &UXGControlFlowsSubsystem::NotifyMainUI,    0.8);
Flow.QueueStep(TEXT("FinishThisInit"),       this, &UXGControlFlowsSubsystem::FinishThisInit,  1.0);

// 3. 执行
Flow.ExecuteFlow();
```

### 5.2 步骤生命周期

每个 QueueStep 绑定的函数签名：

```cpp
void UXGControlFlowsSubsystem::InitXxx(FControlFlowNodeRef SubFlow, double InProgreesValue)
```

- `SubFlow`：当前步骤的控制权引用
  - `SubFlow->ContinueFlow()` — 步骤成功，推进到下一步
  - `SubFlow->CancelFlow()` — 步骤失败，终止整个流程
- `InProgreesValue`：传入的进度值，由 QueueStep 注册时指定

### 5.3 步骤内线程跳跃

每个步骤内部可以自由切换线程，只要最终调用 ContinueFlow/CancelFlow：

```cpp
// 参考：XGControlFlowsSubsystem.cpp

void UXGControlFlowsSubsystem::InitLocalAsset(FControlFlowNodeRef SubFlow, double InProgreesValue)
{
    AsyncTask(ENamedThreads::AnyThread, [this, SubFlow, InProgreesValue]() {
        FPlatformProcess::Sleep(0.2);

        AsyncTask(ENamedThreads::GameThread, [this, SubFlow, InProgreesValue]() {
            CallInitProgress(InitAsyncID, InProgreesValue);
            SubFlow->ContinueFlow();
        });
    });
}
```

**典型的线程跳跃 + Flow 推进模式**：
```
QueueStep → GameThread → AnyThread(工作) → GameThread → ContinueFlow → 下一步
```

## 六、ManageTask 子任务管理（第 18 章）

> ManageTask 是课程作者在 ControlFlows 基础上自行实现的并行子任务管理机制，不属于 UE 原生 API。它解决的是"一个大步骤内需要并行执行 N 个子任务，全部完成后再继续"的问题。

### 6.1 数据结构

```cpp
// 参考：XGControlFlowsSubsystem.h

// 单个子任务
struct FXGControlFlowsTask
{
    FGuid TaskID;
    FGuid BelongToManageTaskID;
    EControlFlowsTaskStatus TaskStatus;  // None / Succeed / Failed
    FString TaskMessage;
};

// 管理任务（N 个子任务的集合）
struct FXGControlFlowsManageTask
{
    FGuid ManageTaskID;
    EControlFlowsManageTaskStatus MangeTaskStatus;  // None / Succeed / Failed / Processing
    int32 TaskNum;               // 子任务总数量
    TArray<TSharedPtr<FXGControlFlowsTask>> Tasks;  // 实际的子任务容器
    FManageTaskResponse ManageTaskResponse;  // 全部完成后的回调
};

// 全局管理列表
TMap<FGuid, TSharedPtr<FXGControlFlowsManageTask>> ManageList;
```

### 6.2 并行执行 + 完成判断算法

```cpp
// 参考：XGControlFlowsSubsystem.cpp

// 创建 ManageTask
TSharedPtr<FXGControlFlowsManageTask> XGManageTaskPtr = MakeShared<FXGControlFlowsManageTask>();
XGManageTaskPtr->TaskNum = 10;  // 预期 10 个子任务
XGManageTaskPtr->ManageTaskResponse.AddLambda([SubFlow, InProgreesValue, this](bool bResult, FString Message) {
    if (bResult)
    {
        CallInitProgress(InitAsyncID, InProgreesValue);
        SubFlow->ContinueFlow();
    }
    else
    {
        CallInitResult(InitAsyncID, bResult, Message);
        SubFlow->CancelFlow();
    }
});

ManageList.Add(XGManageTaskPtr->ManageTaskID, XGManageTaskPtr);

// 派发子任务（异步并行）
for (size_t i = 0; i < 10; i++)
{
    TSharedPtr<FXGControlFlowsTask> TaskPtr = MakeShared<FXGControlFlowsTask>();
    TaskPtr->BelongToManageTaskID = XGManageTaskPtr->ManageTaskID;
    XGManageTaskPtr->Tasks.Add(TaskPtr);

    AsyncTask(ENamedThreads::AnyThread, [this, TaskID, i, ManageTaskID]() {
        FPlatformProcess::Sleep(i % 3);  // 模拟不同耗时的子任务
        // 工作...

        AsyncTask(ENamedThreads::GameThread, [this, TaskID, ManageTaskID, bTaskFinishResult, TaskMessage]() {
            UpdateTaskStatus(TaskID, ManageTaskID, bTaskFinishResult, TaskMessage);
        });
    });
}
```

### 6.3 三段式 Tick 检测（检测→广播→清理）

```cpp
// 参考：XGControlFlowsSubsystem.cpp Tick triad

// 阶段 1：检测所有 ManageTask 的完成状态
for (auto& Tmp : ManageList)
{
    // 子任务数量与预期不匹配 → 继续等待
    if (Tmp.Value->TaskNum != Tmp.Value->Tasks.Num())
        continue;

    // 调用 CheckManageTaskStatus 聚合成最终状态
}

// CheckManageTaskStatus 算法：
// - TaskNum != Tasks.Num() → Processing（子任务还没收齐）
// - 任一 Task 状态为 Failed → Failed
// - 任一 Task 状态为 None → Processing（还有未完成的）
// - 全部 Succeed → Succeed

// 阶段 2：广播完成状态（通过 ManageTaskResponse 回调）
// 阶段 3：从 ManageList 中移除已完成的任务

// 所有对 ManageList 的访问都在 GameThread Tick 中执行
```

### 6.4 线程安全的委托广播

```cpp
// 参考：XGControlFlowsSubsystem.cpp
void UXGControlFlowsSubsystem::CallInitProgress(FGuid InInitAsyncID, double InProgreesValue)
{
    FControlFlowNodeRef CopyFlow = InitProgress;
    AsyncTask(ENamedThreads::GameThread, [TempDelegate, InInitAsyncID, InProgreesValue]() {
        if (TempDelegate.IsBound())
            TempDelegate.Broadcast(InInitAsyncID, InProgreesValue);
    });
}
```

拷贝 delegate 到栈上，再通过 `AsyncTask(GameThread)` 广播，避免跨线程的 delegate 生命周期问题。

## 七、TaskGraph 线程池阻塞陷阱（第 17 章）

TaskGraph 的线程池大小有限（默认通常为 8~16 个 Worker）。如果用 TaskGraph 的任务去**阻塞等待**其他 TaskGraph 任务，会导致线程池耗尽（线程饥饿/Deadlock）。

### 7.1 陷阱模式

> **本质规律**：只要一个 TaskGraph 任务内发生**阻塞等待另一个 TaskGraph 任务**，就可能耗尽线程池。判断时不看 API 名（Async / ParallelFor 等都走 TaskGraph），看是否在 TaskGraph 任务内阻塞等待。

```cpp
// 参考：XGThreadSubsystem.cpp

// 模式 1：FAsyncTask 滥用（模拟大量 AsyncTask 阻塞 2s）
for (int32 i = 0; i < 100; i++)
{
    AsyncTask(ENamedThreads::AnyThread, []() {
        FPlatformProcess::Sleep(2.0);  // 3 个问题于一体
    });
}

// 模式 2：ParallelFor + 长时间 Sleep
ParallelFor(100, [](int32 Index) {
    FPlatformProcess::Sleep(2.0);  // 占用线程池 2 秒
});

// 模式 3：AsyncTask 内嵌 ParallelFor
AsyncTask(ENamedThreads::AnyThread, []() {
    ParallelFor(100, [](int32 Index) {
        FPlatformProcess::Sleep(2.0);
    });
});
```

### 7.2 判断标准

| 条件 | 是否阻塞陷阱 |
|------|------------|
| FRunnable 中 Sleep 或阻塞等待 | **否**（FRunnable 使用 OS 独立线程，不走 TaskGraph 线程池） |
| TaskGraph 任务中 Sleep | **是**（占用线程池 Worker，不让出给其他任务执行） |
| TaskGraph 任务中 Wait() 另一个 TaskGraph 任务 | **是**（两个 Worker 互相等待，导致死锁） |
| TaskGraph 任务中 ParallelFor | **是**（ParallelFor 也吃 TaskGraph Worker） |
| GameThread 上 Wait() TaskGraph 任务 | 否（GameThread 不是 TaskGraph Worker） |

## 八、选型对比

| 维度 | FRunnable | AsyncTask | Async+TFuture | FGraphEvent | ParallelFor | ControlFlows + ManageTask |
|------|-----------|-----------|--------------|-------------|-------------|--------------------------|
| 线程来源 | OS 独立线程 | TaskGraph 线程池 | 可指定（Thread / TaskGraph） | TaskGraph 线程池 | TaskGraph 线程池 | TaskGraph 线程池 |
| 生命周期 | 手动管理（Create → 持有指针 → 等待 → 销毁） | 自动（Fire and forget） | 自动（通过 TFuture 获取结果） | 自动（通过 FGraphEventRef 控制） | 一次调用 | 自动（Flow 级生命周期） |
| 返回值 | 无（通过共享变量回传） | 无 | TFuture\<T\> | 无（通过共享变量回传） | 无（通过闭包引用回传） | 无（通过 ContinueFlow 回调） |
| 任务间依赖 | 自行实现 | 不支持 | 不支持 | 原生支持（Prerequisites） | 不支持 | 原生支持（QueueStep 顺序执行 + ManageTask 并行） |
| 线程安全 | FCriticalSection / FThreadSafeBool | 闭包捕获值拷贝 | 闭包捕获 | 闭包捕获 | FScopeLock | 仅 GameThread 操作 ManageList |
| 阻塞风险 | 无 | 线程池耗尽 | 线程池耗尽（Thread模式下安全） | 线程池耗尽 | 线程池耗尽 | 线程池耗尽 |
| 复杂度 | 高 | 低 | 中 | 中 | 低 | 高 |
| 适用场景 | 长驻后台线程 | 一次性轻量异步 | 需要返回值的异步 | 带依赖链的任务 | 纯计算并行 | 多步骤异步编排 |

**选择流程**：

```
需要长时间独立线程？
  ├─ 是 → FRunnable
  └─ 否 → 需要返回值？
            ├─ 是 → Async + TFuture
            └─ 否 → 需要多步骤编排？
                      ├─ 是 → ControlFlows + ManageTask
                      └─ 否 → 需要任务依赖链？
                                ├─ 是 → FGraphEvent
                                └─ 否 → 需要并行计算？
                                          ├─ 是 → ParallelFor
                                          └─ 否 → AsyncTask
```
