# 依赖注入与配置模式

## 模式全景

UE 没有显式的依赖注入框架，而是通过三种内建机制组合实现了依赖注入与配置管理：

| 机制 | 角色 | 覆盖章节 |
|------|------|---------|
| **Subsystem** | 运行时单例服务注入 | 第13章 + 多章实战 |
| **DeveloperSettings** | 类型化配置注入 | 第15章 |
| **CDO（GetMutableDefault）** | 两种机制的底层对象模型基石 | 第3章 + 第15章 |

三种机制的共同特征：
- 引擎自动管理生命周期，无需手动构造
- 通过反射系统注册，声明即注入
- 全局可访问，消除传参链路

---

## Subsystem：运行时单例服务注入

Subsystem 是 UE 中最接近传统 DI 容器的机制。它不通过构造函数注入依赖，而是通过**引擎自动创建 + 全局查找**的方式提供服务。

### 五种类型与生命周期

| 类型 | 父类 | 生命周期 | 获取方式 | 适用场景 |
|------|------|---------|---------|---------|
| Engine Subsystem | UEngineSubsystem | 引擎启动→关闭 | GEngine->GetSubsystem | 全局独特功能（如 AssetManager） |
| Editor Subsystem | UEditorSubsystem | 编辑器启动→关闭 | GEditor->GetSubsystem | 编辑器工具 |
| GameInstance Subsystem | UGameInstanceSubsystem | GameInstance 创建→销毁 | GameInstance->GetSubsystem | **最常用**，跨关卡持久 |
| World Subsystem | UWorldSubsystem | 关卡加载→卸载 | World->GetSubsystem | 关卡级服务 |
| LocalPlayer Subsystem | ULocalPlayerSubsystem | LocalPlayer 创建→销毁 | LocalPlayer->GetSubsystem | 分屏/本地多人 |

GameInstanceSubsystem 是最常用的类型，跨关卡持久存活。

### 声明模式

```cpp
// XGSimpleSubsystem.h
UCLASS()
class UXGSimpleSubsystem : public UGameInstanceSubsystem, public FTickableGameObject
{
    GENERATED_BODY()
public:
    virtual bool ShouldCreateSubsystem(UObject* Outer) const override;
    virtual void Initialize(FSubsystemCollectionBase& Collection) override;
    virtual void Deinitialize() override;

    virtual void Tick(float DeltaTime) override;
    virtual bool IsTickable() const override;
    virtual TStatId GetStatId() const override;

    UFUNCTION(BlueprintCallable, Category = "XG")
    void AddHealth(int32 InHealthToAdd);

    UPROPERTY(BlueprintAssignable)
    FXGSubsystemActorLocation XGSubsystemActorLocationDelegate;
};
```

**三个生命周期重写点**：

| 重写点 | 作用 | 典型逻辑 |
|--------|------|---------|
| `ShouldCreateSubsystem` | 条件控制是否创建 | 返回 bool，可根据 Outer 判断平台/模式 |
| `Initialize` | 初始化资源 | 保存静态指针、分配资源、绑定回调 |
| `Deinitialize` | 清理资源 | 置空静态指针、释放资源 |

引擎自动调用流程：反射发现 UCLASS → ShouldCreateSubsystem → 创建实例 → Initialize → 运行时使用 → 销毁 → Deinitialize。

### 三种获取方式

```cpp
// 方式一：AActor 直接获取（推荐）
UGameInstance* MyGameInstance = GetGameInstance();
UXGSimpleSubsystem* MySubsystem = MyGameInstance->GetSubsystem<UXGSimpleSubsystem>();

// 方式二：UObject 通过 WorldContext 获取（工具函数用）
UFUNCTION(BlueprintCallable, meta = (WorldContext = "InWorldContextObject"))
void ObjectGetSubsystem(const UObject* InWorldContextObject)
{
    UGameInstance* MyGameInstance = InWorldContextObject->GetWorld()->GetGameInstance();
    UXGSimpleSubsystem* MySubsystem = MyGameInstance->GetSubsystem<UXGSimpleSubsystem>();
}

// 方式三：静态单例（有风险，限生命周期内使用）
static UXGSimpleSubsystem* GetXGSubsystemMyself() { return MySubsystemPtr; }
```

| 方式 | 安全性 | 便捷性 | 适用场景 |
|------|--------|--------|---------|
| AActor 直接获取 | 最高 | 中 | Actor 类中的标准用法 |
| UObject WorldContext | 高 | 低 | BlueprintFunctionLibrary |
| 静态单例 | 低 | 最高 | 快速原型、内部工具 |

**WorldContext 陷阱**：`BlueprintPure` 与 `WorldContext` 元数据组合在打包时会产生编译错误，不要在纯函数中使用 WorldContext。

### Tick 扩展（FTickableGameObject）

UObject 基类没有 Tick 机制。Subsystem 需要额外的 `FTickableGameObject` 接口才能获得 Tick 能力。

```cpp
UCLASS()
class UXGSimpleSubsystem : public UGameInstanceSubsystem, public FTickableGameObject
{
    // 三个必须重写的方法
    virtual void Tick(float DeltaTime) override;
    virtual bool IsTickable() const override;     // return !IsTemplate();
    virtual TStatId GetStatId() const override;    // RETURN_QUICK_DECLARE_CYCLE_STAT
};
```

**bFirstTick 模式**：在 Subsystem 的 `Initialize()` 中某些依赖还未就绪，第一个 Tick 帧是首个安全的操作时机。

```cpp
bool bFirstTick = true;

void UXGSimpleSubsystem::Tick(float DeltaTime)
{
    if (bFirstTick)
    {
        bFirstTick = false;
        // 此时所有 Actor 的 BeginPlay 已执行完毕
    }
}
```

### 事件分发模式

Subsystem 作为事件分发中心，通过 `BlueprintAssignable` 委托将内部状态变化推送给消费者。

```cpp
// 声明委托类型
DECLARE_DYNAMIC_MULTICAST_DELEGATE_TwoParams(FXGSubsystemActorLocation, FString, InActionName, int32, InActionIndexTime);

// Subsystem 中持有委托实例
UPROPERTY(BlueprintAssignable)
FXGSubsystemActorLocation XGSubsystemActorLocationDelegate;

// 触发广播
void UXGSimpleSubsystem::CallLocaion(FString InActionName, int32 InActionIndex)
{
    XGSubsystemActorLocationDelegate.Broadcast(InActionName, InActionIndex);
}
```

这种模式下，Subsystem 不直接调用消费者，消费者在自己方便的时候绑定委托——这是事件驱动的依赖倒置。

## DeveloperSettings：类型化配置注入

DeveloperSettings 是 UE 的类型化配置机制，将配置项映射为 UObject 属性，通过 CDO 机制在运行时注入值。

### 声明模式

```cpp
// 项目级 Settings
UCLASS(Config = XGSampleSettings, defaultconfig)
class UXGSampleSettings : public UDeveloperSettings
{
    GENERATED_BODY()
public:
    virtual FName GetContainerName() const override;   // Project
    virtual FName GetCategoryName() const override;    // XG（侧边栏分类名）
    virtual FName GetSectionName() const override;     // XGSampleSettings（节名）

    UPROPERTY(Config, EditAnywhere, BlueprintReadWrite, Category = "XG")
    FString XGAppKey = TEXT("None");

    static UXGSettings* GetXGXunFeiCoreSettings()
    {
        return GetMutableDefault<UXGSampleSettings>();
    }
};
```

**三个覆盖方法决定 INI 映射路径**：

| 方法 | 返回值示例 | 说明 |
|------|-----------|------|
| `GetContainerName` | `Project` / `Editor` / `Engine` | 顶层分类 |
| `GetCategoryName` | `XG` / `XGPlugins` | Project Settings 侧边栏分组 |
| `GetSectionName` | `XGSampleSettings` | 节名，对应 INI 段名 |

**`UCLASS` 参数说明**：
- `Config = XGSampleSettings` — 决定 INI 段名中的后缀，最终格式 `[/Script/{Module}.{ClassName}]`
- `defaultconfig` — 写入 Default 层级（相对于 Project 层级），对于插件 Settings 用 `defaultconfig` 允许跨项目共享

**`UPROPERTY` 标记说明**：
- `UPROPERTY(Config)` — 标记该属性参与 INI 序列化
- `EditAnywhere` — 可在 Project Settings UI 中编辑
- `VisibleAnywhere` — 只在 UI 中显示，不可编辑（适用于只读信息如版本号）

### 读写模式

**读取**：
```cpp
UXGSampleSettings* Settings = UXGSampleSettings::GetXGXunFeiCoreSettings();
// Settings->XGAppKey 直接使用
```

**写入并持久化**：
```cpp
UXGSampleSettings* Settings = UXGSampleSettings::GetXGXunFeiCoreSettings();
Settings->XGAppKey = InMyAppKey;
Settings->Modify();    // 标记脏
FString ConfigPath = FPaths::ConvertRelativePathToFull(
    FPaths::ProjectConfigDir() / TEXT("DefaultXGSampleSettings.ini"));
Settings->SaveConfig(CPF_Config, *ConfigPath);
```

写入四步流程：获取 CDO → 赋值 → Modify → SaveConfig。`Modify()` 标记对象为脏，确保序列化时写入最新值。

**插件级 Settings 的自动版本号获取**：
```cpp
UCLASS(Config = DataBase, defaultconfig)
class UXGSampleClientSettings : public UDeveloperSettings
{
    UXGSampleClientSettings()
    {
        TSharedPtr<IPlugin> PluginPtr =
            IPluginManager::Get().FindPlugin(TEXT("XGSampleClient"));
        if (PluginPtr.IsValid())
        {
            ClientVersion = PluginPtr->GetDescriptor().VersionName;
        }
    }
};
```

在构造函数中自动获取插件版本号，用户无需手动维护。

### INI 6 层层级

| 层级 | 文件 | 适用场景 |
|------|------|---------|
| Engine Base | Engine/Config/Base.ini | 引擎默认值 |
| Engine Platform | Engine/Config/Base{Platform}.ini | 平台特定引擎默认值 |
| Project Default | Project/Config/Default{Settings}.ini | **开发者编辑的配置** |
| Project Platform | Project/Config/Default{Platform}{Settings}.ini | 平台特定项目配置 |
| Saved/User | Project/Saved/Config/{Platform}/... | 运行时动态生成 |
| Packaged | Pak 中的 INI | 发布版本 |

高层覆盖低层，运行时读取时引擎按层级合并。

### 生成物结构

写入 `DefaultXGSampleSettings.ini` 后内容：
```ini
[/Script/XGSampleDemo.XGSampleSettings]
ProjectSimpleName="1.0.0"
XGAppID="None"
XGAppKey="None"
```

段名格式：`[/Script/{ModuleName}.{ClassName}]`，其中 ModuleName 是 `.Build.cs` 中定义的模块名。

## CDO：两种模式的底层纽带

CDO（Class Default Object）是两种模式的共同基石。无论是 `GetGameInstance()->GetSubsystem<T>()` 还是 `GetMutableDefault<T>()`，底层都依赖 CDO 机制。

### CDO 在两种模式中的作用

| | Subsystem | DeveloperSettings |
|--|-----------|-----------------|
| 实例数量 | 引擎运行时创建 1 个实例 | CDO 本身就是唯一实例 |
| 获取方式 | `GetSubsystem<T>()` | `GetMutableDefault<T>()` |
| 数据来源 | 运行时动态 | INI 文件序列化 |
| 生命周期 | 同 GameInstance | 同模块（全局唯一） |

### GetMutableDefault 原理

`GetMutableDefault<T>()` 返回的是类的 CDO。对于 DeveloperSettings，CDO 本身就是被用作配置存储的对象，修改后调用 `SaveConfig()` 将属性写回 INI。

### UPROPERTY(Config) 标记

`Config` 是标记属性参与 INI 序列化的唯一途径。只有带此标记的属性才会被 `SaveConfig()` 写入、`LoadConfig()` 读取。

## 模式对比表

| 维度 | Subsystem | DeveloperSettings | CDO |
|------|-----------|-----------------|-----|
| **本质** | 运行时单例服务容器 | 类型化配置容器 | UObject 默认对象 |
| **注入方式** | 接口（通过类型查找服务） | 属性赋值（INI → CDO 属性） | 构造时默认值 |
| **生命周期管理** | 引擎自动（条件创建+销毁） | 引擎自动（全局单例） | 引擎自动（UClass 创建时） |
| **变更新通知** | 事件委托 + SelectAsset 回调 | SaveConfig 持久化 + 重启生效 | - |
| **跨关卡持久** | 是（GameInstance 级） | 是（全局） | - |
| **蓝图访问** | GetSubsystem 节点 | GetMutableDefault 节点（Project Settings） | GetClassDefaults |
| **序列化格式** | 无（运行时对象） | INI 文件 | INI/BP 继承 |
| **适用场景** | 需要全局访问的服务 | 需要可视化编辑的配置 | 所有 UObject 的默认值 |

### 选择决策

| 需求 | 选择 |
|------|------|
| 需要全局访问的服务对象（如网络管理器、音频管理器） | Subsystem |
| 需要在 Project Settings 中可视化编辑的配置项 | DeveloperSettings |
| 需要在 C++/蓝图中跨模块存取配置项 | DeveloperSettings |
| 需要在编辑器启动时就存在的工具 | Editor Subsystem |
| 需要 Tick 能力的全局服务 | Subsystem + FTickableGameObject |
| 运行时动态生成的配置（如玩家存档） | 不用这两种，用 SaveGame 或自定义 JSON |
| 仅关卡内存活的临时服务 | World Subsystem |

## 设计原则总结

1. **声明即注入** — Subsystem 和 DeveloperSettings 都只需要 `UCLASS()` 声明，引擎自动管理创建和生命周期，开发者不需要手动实例化
2. **通过类型查找** — UE 的 DI 是"服务定位器"模式（Service Locator），通过类型（UClass）查找服务实例，而非通过构造函数参数注入
3. **全局可见，不需要传参** — Subsystem 和 Settings 全局可访问，消除链式传参，但也会隐藏真实的依赖关系
4. **事件驱动** — Subsystem 通过委托广播状态变化，消费者按需绑定，实现发布-订阅模式的依赖倒置
5. **配置即代码** — DeveloperSettings 让配置项获得类型安全、UI 编辑和序列化支持，配置声明成为模块接口的一部分
