# 插件四层架构设计

## 概述

XGSampleWebP 插件采用**四层封装架构**：原生 C API → 纯 C++ 封装层 → UEC++ 封装层 → Subsystem/BPLibrary 顶层。每一层有明确的职责边界和用户画像。

## 架构全景

```
       蓝图用户 / C++ 用户
              │
    ┌─────────┴──────────┐
    │  BPLibrary + Subsystem  │  ← 第四层：完美封装，一键调用
    │  (UXGSampleWebPBPLibrary)│
    │  (UXGSampleWebPOneShot/   │
    │   MultiShot/ShowMulti)    │
    └─────────┬──────────┘
              │
    ┌─────────┴──────────┐
    │  UEC++ Core 层       │  ← 第三层：UE类型 ↔ C++类型转换
    │  (FXGSampleWebPCore) │     FString→char*, FColor→uint8*
    └─────────┬──────────┘
              │
    ┌─────────┴──────────┐
    │  纯 C++ Lib 层       │  ← 第二层：原子化 API，不依赖 UE
    │  (FXGSampleWebPLib) │     可单独打成 DLL
    └─────────┬──────────┘
              │
    ┌─────────┴──────────┐
    │  原生 libwebp API    │  ← 第一层：Google 提供的 C 接口
    │  (encode.h/decode.h/ │
    │   demux.h/mux.h)     │
    └─────────────────────┘
```

## 各层详解

### 第一层：原生 libwebp API

- 位置：`code/012_第三方库资源/libwebp/` 和插件 `Source/ThirdParty/XGSampleWebPLibrary/`
- 提供头文件：`encode.h`（编码）、`decode.h`（解码）、`demux.h`（解复用）、`mux.h`（复用）、`types.h`（类型定义）
- 静态库：`libwebp.lib`、`libwebpdemux.lib`、`libwebpmux.lib`
- 仅可在 Windows 下使用

### 第二层：纯 C++ 封装（FXGSampleWebPLib）

- 代码：[XGSampleWebPLib.h](file:///d:/UPS/GitLab/XGUnrealNote/Courses/UE-CPP-Basic-to-Advanced/code/001_XGSampleDemo/Plugins/XGSampleWebP/Source/XGSampleWebP/Public/Core/XGSampleWebPLib.h)
- **不依赖 UE 任何类型**，所有参数均为原生 C/C++ 类型（`const char*`、`unsigned char*`、`std::vector`）
- 使用 `friend class FXGSampleWebPCore` 保护内部方法
- 可以独立打成 DLL 分发给非 UE 项目使用

核心 API：
| 方法 | 功能 |
|------|------|
| `GenerateWebpByRGBA` | 单张静态 WebP 编码 |
| `GenerateDymaicWebpByRGBA` | 多张动态 WebP 编码 |
| `LoadDynamicWebpPictureByRGBA` | 动态 WebP 解码 |

### 第三层：UEC++ Core 封装（FXGSampleWebPCore）

- 代码：[XGSampleWebPCore.h](file:///d:/UPS/GitLab/XGUnrealNote/Courses/UE-CPP-Basic-to-Advanced/code/001_XGSampleDemo/Plugins/XGSampleWebP/Source/XGSampleWebP/Public/Core/XGSampleWebPCore.h)
- 职责：**UE 类型 → C++ 类型转化**（`FString` → `const char*`，`TArray<FColor>` → `unsigned char*`）
- 带 `XGSAMPLEWEBP_API` 导出宏，允许用户直接调用
- 包含参数校验（路径合法性、尺寸匹配、质量因子裁剪）
- 标注"外部必须在其他线程使用该接口，否则会阻塞主线程"

### 第四层：BPLibrary + Subsystem 顶层

- BPLibrary：[XGSampleWebPBPLibrary.h](file:///d:/UPS/GitLab/XGUnrealNote/Courses/UE-CPP-Basic-to-Advanced/code/001_XGSampleDemo/Plugins/XGSampleWebP/Source/XGSampleWebP/Public/XGSampleWebPBPLibrary.h) — 蓝图暴露的静态函数
- Subsystem 有三个：
  - `UXGSampleWebPOneShotSubsystem` — 单张 WebP 生成
  - `UXGSampleWebPMultiShotSubsystem` — 多张动态 WebP 生成（继承 `FTickableGameObject`）
  - `UXGSampleWebpShowMultiSubsystem` — 动态 WebP 加载展示（继承 `FTickableGameObject`）
- 所有 Subsystem 内部方法为 `protected`，通过 `friend class UXGSampleWebPBPLibrary` 暴露给 BPLibrary

## 架构设计理由

### 为什么不让用户直接调 Subsystem？

Subsystem 内部涉及复杂的多线程交互流程（截图→回调→异步编码→回主线程→通知蓝图），用户不理解内部机制容易犯错。顶层 BPLibrary 只暴露少量简单接口：

| 蓝图接口 | 功能 |
|---------|------|
| `BeginRecord` / `BeginRecordFullViewport` | 开始记录多帧 |
| `EndRecord` | 结束记录并生成 WebP |
| `LoadWebp` | 加载并展示动态 WebP |
| `ReleaseLoadedWebp` | 释放已加载的 WebP |

### 为什么不只留顶层？

不同级别的用户有不同的需求：

| 用户类型 | 推荐使用层 |
|---------|-----------|
| 蓝图用户 | BPLibrary（第四层） |
| UEC++ 开发者 | FXGSampleWebPCore（第三层） |
| C++ 开发者（非 UE 项目） | FXGSampleWebPLib（第二层，可打成 DLL） |
| 高级 C++ 开发者 | 原生 libwebp API（第一层） |

### 封装粒度的权衡

- 顶层"完美封装"：简单但不可扩展
- 底层"原子化 API"：灵活但需要更多编码能力
- 两层之间可独立打 DLL，适合团队分工（一号团队写 DLL，二号团队写 UE 封装，三号团队写业务逻辑）

## 插件目录结构

```
XGSampleWebP/
├── Source/
│   ├── ThirdParty/XGSampleWebPLibrary/    ← 第一层
│   │   ├── Public/ (encode.h, decode.h, demux.h, mux.h...)
│   │   └── x64/Release/ (libwebp.lib, libwebpdemux.lib, libwebpmux.lib)
│   └── XGSampleWebP/
│       ├── Public/
│       │   ├── Core/XGSampleWebPLib.h     ← 第二层
│       │   ├── Core/XGSampleWebPCore.h    ← 第三层
│       │   ├── Type/XGSampleWebPType.h    ← 数据结构
│       │   ├── Subsystem/...OneShot...h   ← 第四层
│       │   ├── Subsystem/...MultiShot...h
│       │   ├── Subsystem/...ShowMulti...h
│       │   ├── Log/LogXGSampleWebP.h
│       │   └── XGSampleWebPBPLibrary.h    ← 第四层蓝图入口
│       └── Private/ (对应实现)
└── XGSampleWebP.uplugin
```
