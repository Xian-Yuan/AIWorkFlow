# 测试调试与打包修复

## 概述

完成代码编写后进行测试验证，排查常见问题，并确保插件能正确打包。

## 单张 WebP 测试

### 蓝图测试步骤

1. 在关卡蓝图中通过键盘事件触发
2. 获取 `UXGSampleWebPOneShotSubsystem`
3. 传入蓝图动态委托（绑定生成结果回调）
4. 调用 `BeginSampleWebPOneShot`

### 常见问题

#### 问题 1：截图回调未触发

**现象**：日志显示 `bWorking` 被设为 `true` 后，`ScreenShotCallback` 一直未进入。

**原因**：忘记调用 `FScreenshotRequest::RequestScreenshot(false)`。

**修复**：在 `Begin` 流程中，注册截图代理后必须调用 `RequestScreenshot` 主动发起截图。

#### 问题 2：生成的图片全是马赛克

**现象**：可以生成 `.webp` 文件，但打开后图片内容全是乱码色块。

**原因**：A 通道未反转。UE 的 `FColor` A 通道方向与 libwebp 预期相反。

**修复**：在像素转换时遍历每个 `FColor`，执行 `A = 255 - A` 反转。

```cpp
// 在像素转换循环中
FColor ReversedColor = InColor;
ReversedColor.A = 255 - InColor.A;
```

**注意**：反转逻辑在不同场景下可能不同。通过 `Capture2D`、`DrawTarget` 或材质读取时需具体场景验证。

### 调试技巧

- 使用 VS 断点逐步跟踪 `ScreenShotCallback` 是否进入
- 检查 `bWorking` 状态，确保第一次进入时为 `false`
- 代码中保留 `sampleWebp` 前缀的测试日志便于排查

## 多张 WebP 测试

### 蓝图测试步骤

1. 在 Actor 的 `Tick` 或键盘事件中调用 `BeginRecord`
2. 让物体运动（如旋转 Cube）以生成连续变化的帧
3. 若干帧后调用 `EndRecord`，传入完成回调
4. 检查生成的 `.webp` 文件

### 常见问题

#### 问题 1：BeginRecord 时 bBegin 返回 false

**原因**：当前 Subsystem 正在 `Recording` 或 `Generating` 状态。

**修复**：等待上一次流程完成后再调用。

#### 问题 2：生成图片质量问题

**原因**：质量因子设置过低，或者截图期间场景变化过快。

**建议**：使用质量因子 100（最高质量），确保截图对象在录制期间持续运动。

## 打包修复

### 插件打包注意事项

- 第三方库 .lib 文件是 Windows x64 Release 版本，打包时自动包含
- 确保 `XGSampleWebPLibrary.Build.cs` 中的路径使用 `ModuleDirectory` 相对路径
- 如果在打包时遇到"找不到"错误，检查 Build.cs 中的 `PublicIncludePaths` 和 `PublicAdditionalLibraries` 是否使用了绝对路径

### 项目打包

- 依赖 `XGSampleWebP` 插件的项目打包后，插件会自动跟随
- 如果遇到 Link 错误，检查模块依赖链：`项目模块 → XGSampleWebP → XGSampleWebPLibrary`

## 性能建议

- 帧录制期间控制截图频率（不必每帧都截）
- 单张编码 1K 图片约 10~30ms，多张动态图按 N 倍增长
- 超大分辨率截图（2K/4K）需考虑内存占用和编码时间
